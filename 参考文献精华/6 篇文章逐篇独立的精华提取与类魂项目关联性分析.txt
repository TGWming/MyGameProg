## 第 1 篇：《UE4搭建动画树思路》— Janisa

### 精华提取

文章以大逃杀射击游戏为例，提出 **5 层动画树**搭建思路：

```
层级结构：

第1层：基础状态机（按物理状态分类）
  ├── Walking（站/蹲）← BlendSpace 组合方向+速度
  ├── Prone（趴）← 需修改碰撞盒
  ├── Swimming（游泳）
  └── Driving（开车）
  → 通过 BlendPosesBool / 枚举 BlendPoses 切换
  → 输出到 CachedBase

第2层：上半身 Slot 覆盖
  CachedBase → Slot(Upper) → CachedUpper
  上半身动画全部做成 Montage
  Slot 分组：FullBody（全身动画）+ Upper（上半身）

第3层：AimOffset 叠加
  CachedUpper → AimOffset(9方向瞄准) → CachedUpperAim

第4层：LayeredBlendPerBone 上下半身融合
  BasePose = CachedBase（下半身）
  BlendPose = CachedUpperAim（上半身）
  过滤下半身骨骼 → CachedBlend

第5层：FullBody Slot → FinalAnimationPose
  CachedBlend → Slot(FullBody) → Output
  不需要上半身融合的状态（如游泳）用 BlendPosesBool 跳过第4层
```

多武器方案：用武器枚举 BlendPoses 选择不同武器的动画集，再走上述流程。

### 对我们类魂项目的关联性分析

```
| 射击游戏动画树（5层）          | 我们类魂项目（3层）               | 差异原因                    |
|-------------------------------|----------------------------------|---------------------------|
| 第1层：4个基础状态机           | State Machine（7-8个State）       | 类魂无趴/游泳/开车         |
| 第2层：Upper Slot 上半身覆盖   | 不需要                            | 类魂不需要上下半身分离      |
| 第3层：AimOffset 9方向瞄准     | 不需要（P0阶段）                   | 类魂没有枪械瞄准           |
| 第4层：LayeredBlendPerBone     | 不需要                            | 类魂攻击是全身动作          |
| 第5层：FullBody Slot           | FullBody Slot                    | 完全一致                   |
```

**核心结论：**

- 射击游戏需要上下半身分离（边跑边开枪），所以有 5 层
- 我们类魂项目攻击/翻滚/受击都是**全身动作**，不需要分离上下半身
- 我们的结构是：`SM → (Additive Slot) → FullBody Slot → Output`，只有 3 层
- 但文章验证了一个关键设计思路：**CachedPose 缓存节点** 是管理复杂动画树的核心工具，后期如果我们要加弓箭/法杖等需要上半身独立的武器，可以参考第 2~4 层扩展
- 文章的**武器枚举切换思路**可以直接参考：我们锁定/非锁定的 BS 切换用 `BlendPosesBool`，多武器的 Montage 选择用代码枚举

## 第 2 篇：《UE4 动画系统-源码梳理-总览》— 书生

### 精华提取

**核心类字典：**

```
USkeletalMeshComponent    → 骨骼Mesh组件，运行时实例
USkeleton                 → 骨骼类（坐标空间树），Mesh只是一层皮
USkeletalMesh             → 连接Mesh和动画的桥梁（BoneTree + Sockets）
UAnimInstance             → 动画蓝图基类，动画系统核心管理类
FAnimInstanceProxy        → 动画实例代理，支持多线程并行操作
UAnimClassData            → 存放动画蓝图运行时数据（状态机列表/通知列表/节点列表）
```

**动画节点生命周期（每帧执行顺序）：**

```
Initialize_AnyThread  → 节点第一次运行时
CacheBones_AnyThread  → 刷新引用的骨骼索引
Update_AnyThread      → 更新参数（权重/进度/DeltaTime）
Evaluate_AnyThread    → 计算所有骨骼的 Transform → 输出 Pose
GatherDebugData       → ShowDebug Animation 调试用
```

**状态机数据结构：**

```
FBakedAnimationStateMachine
  ├── MachineName                    → 状态机名字
  ├── States[]                       → FBakedAnimationState 数组
  └── Transitions[]                  → FAnimationTransitionBetweenStates 数组

FBakedAnimationState（单个State）
  ├── StateName
  ├── StateRootNodeIndex             → 状态输出节点索引
  ├── Transitions[]                  → 退出转换数组
  ├── StartNotify / EndNotify        → 进入/退出状态的通知
  ├── FullyBlendedNotify             → 完全混合完成的通知
  ├── PlayerNodeIndices[]            → 动画播放节点索引列表
  └── bIsAConduit                    → 是否是导管类型

FAnimationTransitionBetweenStates（转换条件）
  ├── PreviousState / NextState      → 起始/目标状态索引
  ├── CrossfadeDuration              → 过渡混合时长
  ├── StartNotify / EndNotify        → 转换开始/结束通知
  └── InterruptNotify                → 转换被打断的通知
```

**状态机 Update 核心源码流程：**

```
每帧 Update_AnyThread:
  do {
    FindValidTransition()           → 检测退出转换是否满足条件
                                      （委托 > 自动转换 > 蓝图条件）
    如果满足:
      触发当前状态 EndNotify
      触发目标状态 StartNotify
      SetState(NextState)
    TransitionCount++
  } while (有效转换 && Count < MaxTransitionsPerFrame)  // 默认一帧最多3次

  遍历 ActiveTransitionArray:
    如果转换完成:
      触发转换 EndNotify
      触发 FullyBlendedNotify
    如果仍在过渡:
      UpdateTransitionStates()      → 混合两个State输出
  
  如果没有转换:
    只更新当前 State 的输出

  ElapsedTime += DeltaTime
```

### 对我们类魂项目的关联性分析

```
| 源码知识点                    | 对我们的直接价值                                      |
|------------------------------|------------------------------------------------------|
| 节点生命周期 4 函数           | 写自定义 AnimNode 时必须知道（P0不需要，后期可能用到）  |
| 状态机一帧最多转换 3 次       | 跳跃 Idle→JumpStart→InAir 可能在一帧内连续转换，正常  |
| FindValidTransition 判定顺序  | 委托 > 自动转换 > 蓝图条件，我们用蓝图条件（Bool变量）  |
| CrossfadeDuration            | 这就是 State 之间过渡时长，我们设 0.1~0.25 秒          |
| StartNotify / EndNotify      | State 进入/退出时可触发通知，但我们主要用 Montage 通知  |
| FullyBlendedNotify           | 过渡完全完成后触发，可用于确认"已完全进入某状态"        |
| FAnimInstanceProxy 多线程    | 动画变量更新走代理，我们的 SoulsAnimInstance 要用它     |
| InterruptNotify              | 转换被打断时触发，极端情况下的安全网                    |
```

**核心结论：**

- 这篇是**工具书性质**，不需要现在全部掌握，但在以下场景必须查阅：
  - 写 `SoulsAnimInstance` C++ 类时，需要理解 `FAnimInstanceProxy` 的多线程机制
  - 调试 State Machine 过渡异常时，需要理解 `FindValidTransition` 的判定顺序
  - 遇到"过渡闪烁"等 Bug 时，需要理解 `FullyBlendedNotify` 和 `ActiveTransitionArray` 的混合逻辑

## 第 3 篇：《UE4动画系统之叠加动画源码浅析》— hunter

### 精华提取

**叠加动画核心公式：**

```
结果姿势 = 基础原姿势 + (叠加动画姿势 - 叠加Base姿势)
           ↑              ↑ 这就是"增量"
       走路/站着      美术做的叠加资源

本质：减法 + 加法
```

**两种叠加空间：**

```
| 类型                  | 枚举值                          | 用途              | 特征                        |
|----------------------|--------------------------------|-------------------|-----------------------------|
| LocalSpace 叠加      | AAT_LocalSpaceBase             | 局部叠加           | 相对父骨骼方向，随身体朝向变化 |
| MeshSpace 叠加       | AAT_RotationOffsetMeshSpace    | 全局方向叠加        | 相对网格空间，方向固定不变     |
```

**关键区别示例（头部向下叠加）：**

```
身体旋转 90° 时：
  LocalSpace → 头相对 neck_01 向下 → 头朝身体侧面低
  MeshSpace  → 头相对网格空间向下 → 头始终朝世界下方低

选择标准：
  想要局部效果（受击抖动/倾斜）→ LocalSpace
  想要固定方向（端枪上下瞄准）→ MeshSpace
```

**叠加 Base 姿势三种来源：**

```
| 类型              | 枚举值          | 含义                              |
|------------------|----------------|-----------------------------------|
| 参考姿势          | ABPT_RefPose   | 骨骼的默认T-Pose/A-Pose           |
| 整段动画比例       | ABPT_AnimScaled| 当前时间比例映射到Base动画          |
| 指定帧            | ABPT_AnimFrame | 取Base动画的某一帧                 |
```

**常见叠加应用方式：**

```
1. 叠加动画节点
   ApplyAdditive          → 搭配 LocalSpace
   ApplyMeshSpaceAdditive → 搭配 MeshSpace

2. 叠加蒙太奇
   在 Montage 引用的动画上设置叠加类型
   播放时不覆盖基础姿势，而是叠加增量
   例：端枪姿势 + 肩部抖动开火蒙太奇

3. 叠加混合空间
   BlendSpace 内引用的动画设置叠加
   混合后的结果作为增量叠加到基础姿势

4. AimOffset（瞄准偏移）
   本质 = MeshSpace 叠加的 BlendSpace 封装
   引用的动画必须设为 MeshSpace
```

### 对我们类魂项目的关联性分析

```
| 叠加应用场景              | 我们是否需要 | 阶段     | 具体用法                            |
|--------------------------|-------------|---------|-------------------------------------|
| 呼吸/疲劳叠加            | 后期需要     | P1/P2   | Additive Slot 叠加细微呼吸起伏       |
| 受击微抖叠加              | 可能需要     | P1      | LocalSpace 叠加受击时身体微抖         |
| 精力耗尽喘息              | 后期需要     | P1      | LocalSpace 叠加肩部起伏              |
| AimOffset 瞄准偏移        | 不需要       | —       | 类魂没有枪械瞄准                     |
| 叠加蒙太奇（开火抖动）    | 不需要       | —       | 类魂没有射击                         |
| 倾斜叠加（转弯时身体倾斜） | 后期可选     | P2      | 跑步转弯时身体微倾，增加真实感        |
```

**核心结论：**

- P0 阶段**完全不需要叠加动画**，我们的 AnimGraph 中 Additive Slot 先直通
- 后期打磨阶段（P1/P2），叠加动画是提升"活人感"的重要手段
- 最关键的记忆点：**LocalSpace = 随身体方向变化，MeshSpace = 固定世界方向**
- 如果后期做弓箭/法杖武器，AimOffset 这部分知识直接可用

## 第 4 篇：《【GDC精翻】《战神4》战斗系统分享》— 米希尔·谢斯

### 精华提取

**战斗系统三大问题与解决方案：**

```
问题1：找到怪物（Threat Awareness）
问题2：瞄准怪物（Target Selection）
问题3：攻击怪物（Engagement）
```

**1. 仇恨值与发票系统（Aggro & Ticketing）：**

```
定期评估所有怪物，按以下优先级排序：
  1. 怪物是否可以变得有仇恨值（非受击/非死亡状态）
  2. 攻击优先级（设计师设定） × 距离衰减
  3. 是否是玩家当前目标
  4. 动态等级（镜头内外 + 与镜头角度 + 与玩家距离）

排序后从固定池中依序领取攻击票：
  - 大型怪物需要更多票
  - 票用完后，剩余怪物设为无仇恨值
  - 不同难度调整票池大小和怪物攻击性

关键设计：被打断的有仇恨值怪物暂时保留攻击票
  → 给保持进攻的玩家实质性优势
  → 普通难度下同时只有2个怪物能拿票
```

**2. 站位系统（三次迭代）：**

```
第1版：权重网格（失败）
  奎托斯周围生成固定网格 → 多因素加权 → 选最佳位置
  问题：权重复杂/怪物聚集/难调试

第2版：区域约束（改进）
  目标从"选最佳位置"改为"不让怪物进入糟糕位置"
  有仇恨值怪物 → 前方扇形区域（配合镜头视角）
  无仇恨值怪物 → 周围大圆圈
  间距要求：有仇恨值=圆形，无仇恨值=弯曲矩形（横向散布）
  问题：镜头外怪物走到身后成为突发威胁

第3版：象限锁定（最终方案）
  镜头里的怪物保持在镜头里（除非玩家主动转向）
  镜头外的怪物保持在镜头外（除非玩家看向它）
  镜头外的怪物锁定在相对于玩家的世界空间象限内
  → 玩家可建立"心理地图"，知道怪物大概在哪
```

**3. 近战目标选择（多因素加权）：**

```
| 因素                    | 权重作用                              |
|------------------------|---------------------------------------|
| 镜头朝向（右摇杆）      | 主要方向参考                           |
| 是否在屏幕中            | 不在屏幕内直接失效                     |
| 与奎托斯水平角度         | 角度越大权重越低                       |
| 屏幕中心加成            | 靠近屏幕中心权重更高                   |
| 玩家意图（左摇杆）      | 保留传统战神的操作感                   |
| 与奎托斯距离            | 脱离范围直接失效                       |
| 是否是当前攻击目标      | 当前目标有惯性加成                     |

规则：有有效目标时必须选目标（不能打空气）
     无目标时沿镜头方向直接前攻
```

**4. 目标吸附（Suck To Target）：**

```
每次攻击指定：范围 / 速度 / 停止距离
根据目标角度缩放范围：角度越大 → 范围越小
→ 优先攻击屏幕中心的敌人
→ 最高难度大幅缩短吸附距离
```

**5. 击飞辅助（Launch Assist）：**

```
受击怪物如果预期被打出屏幕外 → 轨迹自动修正拉回镜头内
修正量 = 镜头朝向向量 与 原始受击轨迹 的混合
偏差越大 → 修正越大
连续攻击会把目标持续拉向镜头中心

意外收获：
  玩家可以通过控制镜头方向来"辅助瞄准"击飞方向
  → 故意聚怪后释放AoE
  → 用最后一击把怪物打下悬崖
```

**6. 其他辅助系统：**

```
连带受击：受击动画生成隐形碰撞体挤开周围怪物
         被碰到的怪物播放受击但不受伤 → 间接保护玩家

浮空限制：root点高于指定高度时施加向下修正速度
         不依赖重力/物理，保证反复浮空体验一致

镜头外指示器：红色箭头=来袭攻击，白色=空闲怪物，紫色=远程攻击

运镜辅助：横向移动时镜头自动保持威胁在视野中
         攻击时镜头自动重定向到攻击方向
```

### 对我们类魂项目的关联性分析

```
| 战神4系统                | 类魂是否需要 | 优先级 | 说明                                    |
|-------------------------|-------------|--------|----------------------------------------|
| 仇恨值 & 发票系统        | ✅ 需要      | P1     | 控制同时攻击的敌人数量，核心战斗节奏      |
| 站位系统（区域约束）      | ✅ 需要      | P1     | 防止怪物堆叠，保持战场可读性              |
| 象限锁定                 | 可选         | P2     | 类魂锁定系统已解决部分问题                |
| 近战目标选择              | ✅ 需要      | P0     | 非锁定下攻击朝向哪个敌人，核心手感        |
| 目标吸附                 | ✅ 需要      | P0     | 类魂攻击也有前冲位移，需要对准目标        |
| 击飞辅助                 | 不需要       | —      | 类魂受击位移很小，不存在打飞出屏幕问题     |
| 连带受击                 | 可选         | P2     | 类魂群战较少，但可用于大型武器横扫         |
| 浮空限制                 | 不需要       | —      | 类魂没有浮空连招                         |
| 镜头外指示器              | 不需要       | —      | 类魂有锁定系统，且战斗节奏较慢            |
| 运镜辅助                 | 可选         | P2     | 类魂玩家习惯自己控制镜头                  |
```

**核心结论：**

- 这篇是**战斗设计层面最有价值**的文章，直接指导敌人 AI 行为设计
- P0 阶段最需要的：**近战目标选择 + 目标吸附**（决定攻击手感）
- P1 阶段最需要的：**仇恨值发票系统 + 站位系统**（决定多敌人战斗体验）
- 击飞辅助虽然我们不需要，但其设计思路（"修正不合理的物理结果以配合镜头"）在调受击方向时可借鉴

## 第 5 篇：《UE4/UE5 动画的原理和性能优化》— quabqi

### 精华提取

**骨骼动画两步核心流程：**

```
第1步：GameThread → TickComponent → 求得当前帧最终 Pose
  内部细分：
  ├── 更新动画蓝图变量
  ├── UpdateAnimation（更新进度/算权重/算执行路径）→ 可子线程
  ├── EvaluateAnimation（解算骨骼 Transform）→ 可子线程
  └── 通知 Notify/NotifyState → 必须 GameThread

第2步：RenderThread → 根据 Pose 做蒙皮 → 绘制
  ├── GPUSkin → Shader 里蒙皮（推荐）
  ├── CPUSkin → 渲染线程 C++ 蒙皮
  └── Static  → 当 StaticMesh 绘制（无动画）
```

**蒙皮本质：**

```
最终顶点位置 = Σ (骨骼Transform × 该骨骼对该顶点的权重)

GPU Shader 实现（GpuSkinVertexFactory.ush）：
  CalcBoneMatrix() → 遍历关联骨骼 → 矩阵×权重求和
  最终顶点 = 顶点位置 × CalcBoneMatrix结果
```

**12 条优化手段：**

```
| #  | 优化手段                       | 效果       | 阶段     | 对类魂项目适用性           |
|----|-------------------------------|-----------|---------|--------------------------|
| 1  | Update/Evaluate 放子线程       | ★★★★★    | P0就做   | 必须做，直接减GameThread   |
| 2  | 减少 Notify/NotifyState        | ★★★      | 中后期   | 注意不要滥用Notify         |
| 3  | 避免 RootMotion                | ★★★      | 设计期   | 类魂翻滚用RootMotion，需权衡|
| 4  | 开启 URO（跳帧）               | ★★★★★    | P0就做   | 远处NPC/敌人必须开         |
| 5  | 切换 TickPose 模式             | ★★★★     | P0就做   | 不渲染时降级               |
| 6  | CopyPose 共享动画              | ★★★      | 后期     | 大量同类NPC时使用          |
| 7  | 简化 AnimBP 执行路径           | ★★★★     | 设计期   | 我们7-8个State已经很简      |
| 8  | 烘焙顶点动画                   | ★★       | 特殊场景  | 远景人群可用               |
| 9  | 减骨骼减面                     | ★★       | 资源期   | 不要做千足蜈蚣就行          |
| 10 | 异步加载动态资源               | ★★★      | P0就做   | 动态换武器/换装时用         |
| 11 | 延迟 Initialize                | ★★★      | 需改源码  | 可选，跑不到的节点零开销    |
| 12 | Mesh 合并换装                  | ★★★      | 后期     | 换装系统时用               |
```

### 对我们类魂项目的关联性分析

```
P0 阶段必须做的优化：
  ├── #1 子线程 Update/Evaluate（AnimBP 设置里开启）
  ├── #4 URO 跳帧（配合动画预算分配器）
  ├── #5 不渲染时降低 TickPose 模式
  └── #7 保持 AnimBP 简洁（我们已经只有 7-8 个 State）

需要注意的设计决策：
  #3 RootMotion 问题：
    我们的翻滚用 RootMotion 驱动位移
    RootMotion 可能影响子线程执行
    解决方案：只在 Montage 里用 RootMotion，SM 内不用
    文章建议"最多只用蒙太奇的 RootMotion"——与我们方案一致

后期优化阶段参考：
  ├── #6 CopyPose（大量同类敌人时）
  ├── #10 异步加载（换武器时）
```

**核心结论：**

- 这篇是**性能优化的核心参考**，GameThread 动画开销占比可达 1/2 到 2/3
- 最重要的认知：**动画优化 = 优化 TickComponent**，而 TickComponent = Update + Evaluate + Notify
- 我们 P0 阶段 AnimBP 已经很简洁（7-8 State + Montage 覆盖），先天优势
- RootMotion 只在 Montage（翻滚）里用，完全符合文章建议

## 第 6 篇：《GDC—《星球大战绝地》的物理动画》— Captain

### 精华提取

**三种物理驱动方式：**

```
| 方式          | 原理              | 优点                    | 缺点                  | 适用场景           |
|--------------|-------------------|------------------------|-----------------------|-------------------|
| Motor（力）   | 关节局部力驱动     | 自然流畅/支持碰撞       | 调参复杂/可能摆动过大   | 小怪/悬挂/攀爬     |
| Velocity（速度）| 计算所需速度设置  | 稳定/可控              | 不够自然               | 稳定运动/hip控制   |
| Constraint（约束）| 位移约束绑定   | 精确限制/防止偏移过大    | 大偏差时看起来不自然    | 手部固定/机器人    |
```

**死亡动画方案（关键参考）：**

```
基础做法：动画最后阶段启动物理 → 自由落体
问题：靠墙时动画播放期间就穿墙了

改进方案：
  1. 所有关节设置 Motor（物理驱动贴合动画）
  2. hip 不能是 kinematic（否则带动整体穿墙）
  3. hip 用新约束绑定到动画体位置（线位移+角位移）
  4. 监控 hip 物理体与动画体的距离
     → 距离过大 = 碰墙了 → 切换到自由落体模式（清除之前速度）
  
  流程：Motor驱动全身 → hip约束跟随动画 → 碰墙检测 → 自由落体
```

**物理与动画混合：**

```
在骨骼 Local Space 下混合 AnimPose 和 PhysicsPose
混合比例可调（如滑动场景用 50:50）
弱Motor = 流畅有弹性但摆动多
强Motor = 稳定但僵硬
混合 = 给动画师更多控制空间
```

**BD 伙伴机器人方案：**

```
腿底部 kinematic → 其他部分 100% 物理
头部约束到动画体，动态调整距离限制：
  慢速 → 允许大摆动（自然感）
  快速 → 距离限制趋近0（稳定性）
  极快 → 锁定模式

独立物理场景：
  不需要碰撞的物理动画放在单独的 Physics Scene
  固定帧率 + 可控执行顺序
```

**PhysX 调参要点：**

```
Solver Iteration: 64 position + 32 velocity（他们的配置）
降低摩擦力和恢复系数（否则物理难以驱动动画体）
连续碰撞检测(CCD)：效果好，但不要在同一布偶相连关节启用
约束投影：CCD开启后不兼容，需自己写线性投影
```

### 对我们类魂项目的关联性分析

```
| 物理动画应用            | 类魂是否需要 | 优先级 | 说明                                     |
|------------------------|-------------|--------|----------------------------------------|
| 死亡动画物理方案         | ✅ 后期需要  | P1/P2  | 死亡后布娃娃 + 防穿墙，直接可参考         |
| Motor驱动小怪           | 不需要       | —      | 类魂小怪用固定受击动画，不需要物理驱动      |
| 悬挂/攀爬物理           | 不需要       | —      | 类魂P0没有攀爬系统                        |
| 物理与动画混合           | 可选         | P2     | 后期打磨受击/死亡的物理真实感              |
| 伙伴机器人方案           | 不需要       | —      | 类魂没有跟随伙伴                          |
| 速度驱动稳定hip          | 可选         | P2     | 特殊动画过渡时减少抖动                    |
```

**核心结论：**

- P0 阶段**完全不需要物理动画**，我们的死亡用 Montage + Dead State 静态 Pose
- P1/P2 阶段如果要做**死亡后布娃娃倒地**，这篇的方案（Motor + hip 约束 + 距离监控 → 自由落体）是最佳参考
- 关键认知：物理动画的本质是**用物理过程来补充动画帧之间的运动**，让结果更自然
- Motor 力度的平衡（太弱摆动/太强僵硬）是所有物理动画的核心调参难点
