# 🔍 综合差异分析报告

## 总纲：两套设计体系的核心哲学差异

在逐项比对之前，必须先点明一个**根本性的架构分歧**，它贯穿了所有细节差异：

| 维度 | 文档 C（选项C）的路线 | 文档 3.txt（动画状态机设计表）的路线 |
|---|---|---|
| **状态机定位** | SM 只做 Locomotion 基底（Idle/Move/InAir/Dead），一切动作走 **Montage + FullBody Slot 覆盖** | SM 做 **全量状态承载**，7个状态机 × 90个 State 全部在 SM 内管理 |
| **AnimBP 复杂度** | AnimGraph 极简：`MainSM → Additive Slot → FullBody Slot → Output` | AnimGraph 极重：7 个并行 SM + 优先级总控 + Blend Poses by Bool 逐层切换 |
| **C++ 控制权** | C++ 侧（CombatComponent）拥有几乎全部动作决策权，AnimBP 只"读变量选动画" | AnimBP 侧承担大量过渡判断，C++ 侧需要维护 44 个 Bool + 9 个 Enum |
| **变量规模** | ~20 个核心变量（8 ��举 + 6 浮点 + 4 内部 Bool） | 44 Bool + 16 Float + 5 Int + 9 Enum = **74+ 变量** |

**结论：两套方案在架构哲学上是对立的。**  
文档 C 是 **"Montage 驱动，SM 做基底"**；  
文档 3.txt 是 **"SM 驱动，Montage 建议但不强制"**。

下面逐层展开具体差异。

---

## 第一层：状态机结构差异

### 1.1 状态机数量与职责

| 状态机 | 文档 C | 文档 3.txt | 差异 |
|---|---|---|---|
| 基础移动 | `MainSM > Grounded > Idle/Movement/TurnInPlace` | `SM_Locomotion_Base`（9 States） | 3.txt 多了 `StartMove / Sprint / JumpStart / InAir / Land / Stop` 在 Base 内 |
| 战斗移动 | 无独立 SM；Movement 内用 `MovementStyle` 切 BS | `SM_Locomotion_Combat`（6 States） | **文档 C 没有独立战斗移动 SM** |
| 动作 | 无 SM；全走 Montage Slot | `SM_Action`（26 States） | **根本性差异：C 无此 SM** |
| 交互 | 无 SM；走 Montage | `SM_Interaction`（15 States） | **根本性差异：C 无此 SM** |
| 梯子 | 无 SM；后期扩展提到 Ladder BS | `SM_Ladder`（8 States） | **根本性差异：C 无此 SM** |
| 受击 | `HitReact_Gate`（门控空壳，Montage 覆盖） | `SM_HitReact`（12 States） | C 只有 1 个门控 State，3.txt 有 12 个完整 State |
| 死亡 | `Dead`（1 State，静态 Pose） | `SM_Death`（4 States） | C 只有 1 State，3.txt 拆了 Normal/Fall/Special/DeadLoop |

**总计**：文档 C = **约 6 个 State**；文档 3.txt = **约 90 个 State**。

### 1.2 关键矛盾点

**矛盾 1：Attack Combo 在哪管理？**

| 文档 C 方案 | 文档 3.txt 方案 |
|---|---|
| CombatComponent 用 `Montage_JumpToSection` 在**同一个 Montage 的 Section** 间跳转，SM 完全不参与 | SM_Action 内建 `LightAttack_01 → 02 → 03`，每段是独立 State，用 `bCanCombo && ComboIndex` 做过渡条件 |

> 这是最大的实现路线分歧。文档 C 的方式更主流、更灵活（DS3 实际也是 Montage 驱动），但 3.txt 明确要求了 SM 版本。

**矛盾 2：受击是门控 Pose 还是完整状态机？**

| 文档 C 方案 | 文档 3.txt 方案 |
|---|---|
| `HitReact_Gate` 输出一个静态 Combat Idle Pose，受击 Montage 通过 FullBody Slot 覆盖 | `SM_HitReact` 有 12 个 State（Light/Medium/Heavy/GuardImpact/GuardBreak/Stagger/Knockdown/GetUp/BackstabVictim/RiposteVictim/AirHit），每个 State 绑独立动画 |

> 如果走 3.txt 路线，AnimBP 内部需要处理 `HitReactType` × 方向的组合逻辑，复杂度远高于 C 的门控方案。

**矛盾 3：基础移动 vs 战斗移动是否拆分？**

| 文档 C 方案 | 文档 3.txt 方案 |
|---|---|
| 同一个 `Movement` State 内，用 `MovementStyle` 枚举（FreeMove / LockOnStrafe）选不同 BS | 独立两个 SM：`SM_Locomotion_Base` 和 `SM_Locomotion_Combat`，各有自己的 Idle/Move/Sprint/Guard 状态 |

> 3.txt 的拆分更贴近 DS3 的实际表现（战斗待机和探索待机是完全不同的姿态树），但维护成本翻倍。

| `EHitReactState` (None/LightHit/MediumHit/HeavyHit/Stagger/Knockdown/GetUp/Death) | `EHitReactType`（11 值）类似 | 近似一致，3.txt 多了 `GuardImpact / AirHit / BackstabVictim / RiposteVictim` |

| — | `EDodgeDirection` (None/Forward/Backward/Left/Right) | C 无此枚举 |
| — | `ELadderEntryType` (None/Top/Bottom) | C 无此枚举 |
| — | `EInteractionType`（15 值） | C 无此枚举 |

## 第三层：AnimGraph 连接结构差异

### 文档 C 的 AnimGraph（P0）

```
[MainStateMachine] → [Additive Slot] → [FullBody Slot] → [Output Pose]
      │                    │                   │
      ├── Grounded         │(直通)             │ 所有 Montage 在这里覆盖
      │   ├── Idle         │                   │
      │   ├── Movement     │                   │
      │   └── TurnInPlace  │                   │
      ├── InAir            │                   │
      ├── HitReact_Gate    │                   │
      └── Dead             │                   │
```

### 文档 3.txt 的 AnimGraph（推荐）

```
优先级总控表（Blend Poses by Bool 链 或 Select by Priority）：

Priority 1: bDead → SM_Death
Priority 2: bExecuted || bBackstabbed → SM_HitReact
Priority 3: bHitReact || bGuardBreak || bKnockdown → SM_HitReact
Priority 4: bIsOnLadder → SM_Ladder
Priority 5: bInteracting → SM_Interaction
Priority 6: bInActionState → SM_Action
Priority 7: bCombatMode || bLockOn || bGuarding → SM_Locomotion_Combat
Priority 8: Default → SM_Locomotion_Base

每个 SM 输出 → Cache Pose → 进入优先级选择器 → Overlay Layer → Slot → Output
```

同时还有 Overlay 层：

```
UpperBody_Overlay (Layered Blend Per Bone)
Slot_FullBody
Slot_UpperBody
Additive_Aim (AimOffset)
Additive_IdleBreak
```

> **AnimGraph 复杂度差异巨大。**  
> 文档 C 是 3 个节点的线性链；  
> 文档 3.txt 是 7 个 SM + 8 级优先级 + 5 个 Overlay/Slot 层。

### 5.2 CombatComponent 的覆盖范围

文档 C 的 CombatComponent 包揽了：
- TryDoAction 动作请求
- 连段逻辑（JumpToSection）
- 输入缓存
- 受击处理（方向判断 + Montage 选择 + 播放）
- 无敌帧 / 霸体管理
- 耐力联动
- 委托广播

**而文档 3.txt 没有设计任何 C++ 组件**，它假设这些逻辑分散在 AnimBP 的过渡条件和 State 内部。这意味着如果走 3.txt 路线：

- 连段窗口判断在 AnimBP 的 State 过渡条件中（`bCanCombo && ComboIndex == 2`）
- 受击类型选择在 AnimBP 的 State 过渡条件中（`HitReactType == Light`）
- 交互类型选择在 AnimBP 的 State 过渡条件中（`InteractionType == OpenDoor`）

> 这正是文档 C 第十一章陷阱 1 所警告的：**"在 AnimBP Event Graph 中写复杂逻辑"**。

## 核心区别：谁决定"现在播什么动画"？

### 方案一：Montage 驱动（文档 C 的路线）

**AnimBP ���管"走路站着飞着"这几件事。**

```
AnimBP 的状态机里只有：
  站着（Idle）
  走路（Movement）
  空中（InAir）
  死了（Dead）

就这么几个。
```

**攻击、翻滚、喝药、受击、开门……全部不在状态机里。**

那这些动画怎么播？

→ **C++ 代码直接调用 `Montage_Play()`**，强行把动画"盖"在状态机输出上面。

打个比方：

> 状态机就像一台**一直在播的电视**，永远在放"走路/站着"的节目。
> 
> 当你按下攻击键，C++ 代码就像拿了一张**不透明的海报**糊在电视屏幕上——你看到的是攻击动画（海报），但电视（状态机）其实还在后面默默播走路。
> 
> 攻击动画播完，海报撕掉，你又看到电视了（回到走路/站着）。

这张"海报"就是 **Montage**，"糊上去"的动作就是 **Slot 覆盖**。

---

### 方案二：SM 全量承载（3.txt 的路线）

**AnimBP 的状态机管所有事。**

```
AnimBP 的状态机里有：
  站着、走路、冲刺、跳起、空中、落地、
  战斗站着、战斗走路、举盾站着、举盾走路、
  轻击第1段、轻击第2段、轻击第3段、重击、蓄力、
  前滚、后滚、左滚、右滚、后撤步、
  弹反、战技起手、战技派生、施法起手、施法释放、
  喝药、用道具、拉弓、射箭、
  拾取、开门、拉杆、开宝箱、穿雾门、点篝火、坐篝火、
  爬梯子上、爬梯子下、滑梯子、
  轻受击、中受击、重受击、破防、击倒、起身、被背刺、被处决、
  普通死亡、坠落死亡、特殊死亡……
  
  总共90个状态。
```

每两个状态之间要写**过渡条件**（什么时候从A切到B）。

打个比方：

> 状态机就像一个**超级复杂的流程图**，90个方框，几百条连线。
> 
> 按下攻击键 → AnimBP 里的某个 Bool 变成 true → 流程图顺着箭头从"战斗站着"走到"轻击第1段" → 动画开始播 → 播完了条件变了 → 流程图又顺着箭头走到下一个方框。
> 
> **所有事情都在这张流程图里发生。**

## 对比总结

| | Montage 驱动 | SM 全量 |
|---|---|---|
| **AnimBP 状态机里有什么** | 只有走路/站着/空中/死亡（~6个状态） | 所有动作全在里面（~90个状态） |

这就是两者最本质的区别。





 **Montage 的 Blend 效果并不差**，甚至可以说很好。原因如下：

---

## Montage 并不是"硬切"

你可能想象的是这样：

```
走路动画在播 ████████████
                          ↓ 突然一刀切断
攻击动画开始播             ████████████
```

但实际上 Montage 是这样的：

```
走路动画在播 ████████████████
                     ╲  BlendIn（渐入）
攻击动画开始播          ▓▓████████████
                                  ╲  BlendOut（渐出）
走路动画恢复                         ▓▓█████████��██
```

**每个 Montage 都有两个关键参数：**

| 参数 | 作用 | 典型值 |
|---|---|---|
| **BlendIn** | 从当前动画**渐变**到 Montage 动画，不是硬切 | 0.1 ~ 0.25 秒 |
| **BlendOut** | 从 Montage 动画**渐变**回状态机的动画 | 0.15 ~ 0.3 秒 |

这个渐变就是你说的 Blend。**它和 State Machine 里两个 State 之间的过渡 Blend 是完全一样的底层机制。**

---

## 打个比方

| State Machine 的 Blend | Montage 的 Blend |
|---|---|
| 你从"走路"State 过渡到"攻击"State，UE 在 0.2 秒内把两个动画混合 | 你在"走路"上面贴了一个 Montage，UE 在 0.2 秒内把走路混合到攻击 |
| **底层用的是同一个混合算法** | **底层用的是同一个混合���法** |

所以 Blend 质量是**一模一样的**。

---

## 那为什么魂like游戏的过渡看起来这么干脆利落？

因为**这类游戏故意把 BlendIn 设得很短**。

这不是技术限制，而是**设计选择**：

| 游戏类型 | BlendIn 时间 | 手感 |
|---|---|---|
| 黑魂3 / 血源 / 匹诺曹 | **0.05 ~ 0.15 秒**（很短） | 按键后角色**立刻**进入动作，手感干脆 |
| 刺客信条 / 巫师3 | **0.2 ~ 0.4 秒**（较长） | 动画过渡丝滑，但操作感觉"黏" |
| 格斗游戏 | **接近 0 秒** | 几乎硬切，帧数精确 |

> 魂like 的"沉重感"和"即时反馈感"，恰恰来自于 **BlendIn 短 + 动画前摇长**。
> 
> 不是 Blend 差，而是**故意 Blend 得快**。

---

## Montage 还能做 SM 做不到的 Blend 技巧

### 技巧 1：BlendIn Curve（混合曲线）

```
普通线性 Blend：  走路 ████▓▓▓▓ 攻击
                        均匀过渡

自定义曲线 Blend：走路 ████▓█ 攻击
                        快速切上半身，下半身延迟跟上
```

Montage 可以设置 **Blend Curve**（线性、缓入缓出、自定义曲线），SM 的 State 过渡也能设但没那么灵活。

### 技巧 2：分层 Blend（上下半身分开）

```
下半身：还在播走路 ████████████████████
上半身：Montage 覆盖攻击动画 ████████████
```

比如 **边跑边喝药**：
- 下半身继续走路（State Machine 输出）
- 上半身播喝药动画（Montage 通过 UpperBody Slot 覆盖）

这在 SM 里要做的话，你需要把"走路+喝药"做成一个单独的 State，**每种组合都要一个新 State**。

### 技巧 3：Montage 之间的切换

```
轻击第1段 Montage 还在播 ████████████
                              ↓ BlendIn 0.1s
轻击第2段 Montage 开始播       ▓▓████████████
```

连段之间的 Montage 切换也是带 Blend 的，不是硬切。

## 实际对比

| 场景 | SM 做法 | Montage 做法 | Blend 效果 |
|---|---|---|---|
| 站着 → 攻击 | Idle State → Attack State，过渡时间 0.1s | Idle 上面贴 Attack Montage，BlendIn 0.1s | **一样** |
| 攻击 → 站着 | Attack State → Idle State，过渡时间 0.2s | Attack Montage 结束，BlendOut 0.2s 回到 SM 的 Idle | **一样** |
| 走路 → 翻滚 | Move State → Roll State，过渡时间 0.1s | Move 上面贴 Roll Montage，BlendIn 0.1s | **一样** |
| 攻击第1段 → 第2段 | State 过渡 0.05s | JumpToSection 或新 Montage BlendIn 0.05s | **一样** |
| 跑步 + 喝药（上下分离） | 需要专门做一个"跑步喝药"State 或额外混合逻辑 | UpperBody Slot 播 Montage，下半身不受影响 | **Montage 更好** |

## 一句话总结

> **Montage 的 Blend 和 State Machine 的 Blend 用的是同一套底层系统，效果完全一样。**
> 
> Montage 不是"硬贴海报"，而是"带淡入淡出的透明海报"。
> 
> 魂like 游戏过渡看起来"快"，不是因为 Blend 差，而是**故意设计成短 Blend + 长前摇**来制造沉重感和操作精确感。





先纠正一个关键点

**翻滚和跳跃不是 BlendSpace，它们是 Montage（或单个动画）。**

| 动作 | 用什么 | 原因 |
|---|---|---|
| 前后左右走/跑 | BlendSpace | 需要根据速度和方向**连续混合**多个动画 |
| 翻滚 | Montage | 固定动画，播一次就完，不需要混合 |
| 跳跃 | 单个动画 / Montage | 固定动画，播一次 |
| 落地 | 单个动画 | 固定动画 |

**BlendSpace 只用在"需要连续变化"的场景。**

比如走路：你推摇杆推一点点是慢走，推到底是跑，朝左推是左走，朝右是右走。这个"一点点到到底"之间是**无级变速**的，所以需要 BlendSpace 根据数字连续混合。

翻滚就是翻滚，不存在"翻一半"或者"慢慢翻"，所以不需要 BlendSpace。

```
State Machine 里的每个 State（方框），里面装的东西不一样：

┌──────────────────────────────────────────────────┐

│                                                   │
│  ┌────────┐        ┌────────────┐                │

│  │        │        │            │                │
│  │ 里面装：│        │ 里面装：    │                │
│  │ 一个    │        │ 一个       │                │
│  │ 动画    │        │ BlendSpace │                │
│  └────────┘        └────────────┘                │
│       ↕                                           │
│  ┌────────┐        ┌────────────┐                │
│  │ InAir  │ ←────→ │   Land     │                │
│  │        │        │            │                │
│  │ 里面装：│        │ 里面装：    │                │
│  │ 一个    │        │ 一个       │                │
│  │ 动画    │        │ 动画       │                │
│  └────────┘        └────────────┘                │
│                                                   │
└──────────────────────────────────────────────────┘
```

注意看：
- **Idle** 里面装的是一个**单独的动画**（站着不动）
- **Movement** 里面装的是一个 **BlendSpace**（根据 Speed/Direction 混合走/跑/各方向）
- **InAir** 里面装的是一个**单独的动画**（下落循环）
- **Land** 里面装的是一个**单独的动画**（落地恢复）

**State Machine 不是���"多个 BlendSpace"串起来。** 它是把"不同的东西"（有的是单个动画，有的是 BlendSpace）串起来。

## 代码做了什么？State Machine 做了什么？

这是你问题的核心，我用一个完整的例子说清楚。

### 场景：玩家从站着 → 开始跑 → 跑起来 → 停下来

**代码每帧做的事（C++ Tick）：**

```
每一帧：
  从引擎获取角色当前速度 → 算出 Speed = 356.0
  从输入获取摇杆方向     → 算出 Direction = -30.0（偏左前方）
  判断是否有输入         → bShouldMove = true
  判断是否在地上         → bIsFalling = false

  就这些。代码只算数字，不管动画。
```

**State Machine 每帧做的事（AnimBP）：**

```
每一帧：
  读取 bShouldMove 和 bIsFalling

  当前在 Idle State
  检查过渡条件：bShouldMove == true？ → 是！
  → 开始过渡到 Movement State（0.2秒 Blend）

  现在在 Movement State
  Movement State 里面装着 BS_Locomotion_2D
  这个 BlendSpace 读取 Speed=356.0 和 Direction=-30.0
  → 自动混合出"偏左前方跑步"的动画
  
  下一帧 Speed=360.0，Direction=-28.0
  → BlendSpace 自动更新混合结果
  
  下一帧 Speed=0，bShouldMove=false
  检查过渡条件：bShouldMove == false？ → 是！
  → 开始过渡回 Idle State（0.2秒 Blend）
```

### 所以职责���这样分的：

| 谁 | 做什么 |
|---|---|
| **C++ 代码** | 每帧算出 `Speed / Direction / bShouldMove / bIsFalling` 这些数字和布尔值 |
| **State Machine** | 根据布尔值决定**当前在哪个 State**（Idle？Movement？InAir？），并处理 State 之间的过渡 Blend |
| **BlendSpace** | 坐在 Movement State 里面，根据 `Speed` 和 `Direction` 这两个数字**自动混合**出具体的动画姿势 |

```
代码算数字 → State Machine 选方框 → 方框里的 BlendSpace 混动画
   ↑              ↑                        ↑
 每帧更新      根据条件切换            根据数字自动混合
```

## 那"锁定移动"和"普通移动"怎么切换？

这里你说的"代码切换用哪个 BlendSpace"是对的，但切换方式不是代码直接操作 BlendSpace，而是通过变量让 State Machine 内部选：

```
Movement State 内部：

  读取 bIsLockedOn

  bIsLockedOn == false？
  → 用 BS_BaseLocomotion_2D（Speed + Direction）
     普通移动，角色朝移动方向转身

  bIsLockedOn == true？
  → 用 BS_CombatLocomotion_2D（ForwardAxis + RightAxis）
     锁定移动，角色面朝敌人，八方向平移
```

代码只负责设置 `bIsLockedOn = true / false`，AnimBP 里用一个 `Blend Poses by Bool` 节点选用哪个 BlendSpace。

跳跃这个问题很好，但答案和你想的不太一样。

---

## 跳跃不用 Montage，它适合放在 State Machine 里

原因很简单：

| 特征 | 攻击/翻滚/喝药 | 跳跃的4个阶段 |
|---|---|---|
| 什么时候发生 | 玩家按键，播一次就完 | 角色离地 → 自动经历全过程 → 落地 |
| 和移动的关系 | 打断移动，播完回到移动 | **本身就是移动的一部分** |
| 需要每帧更新吗 | 不需要，播就行了 | 需要，空中时间不固定，落地时机不固定 |
| 持续时间 | 固定的（动画多长就多长） | **不固定**（跳下悬崖可能飘5秒，跳一个台阶只飘0.3秒） |

**关键点：空中阶段的持续时间是不确定的。**

如果用 Montage，你就要播一个固定长度的动画。但玩家从悬崖跳下去飘了5秒怎么办？Montage 播完了角色还在空中，就穿帮了。

而 State Machine 的 `InAir` State 可以放一个**循环动画**，不管飘多久都一直循环，直到条件变了（落地了）才过渡出去。

---

## 跳跃在 State Machine 里长这样

```
State Machine:

┌────────┐     ┌───────────┐     ┌────────┐     ┌────────┐
│  Idle  │     │ JumpStart │     │ InAir  │     │  Land  │
│  站着  │     │   起跳     │     │  空中   │     │  落地  │
│        │     │           │     │        │     │        │
│ 单个   │     │ 单个      │     │ 单个   │     │ 单个   │
│ 动画   │     │ 动画      │     │ 循环   │     │ 动画   │
│        │     │(播一次)    │     │ 动画   │     │(播一次) │
└───┬────┘     └─────┬─────┘     └───┬────┘     └───┬────┘
    │                │                │               │
    │  bIsFalling    │  动画快播完     │  !bIsFalling   │  动画播完
    │  == true       │  或已离地       │  == true       │  
    ├───────────────→├───────────────→├──────────────→├──────→ 回到 Idle
    │                                                         或 Movement
```

每个阶段：

| State | 里面装什么 | 过渡到下一个的条件 | 为什么不用 Montage |
|---|---|---|---|
| **JumpStart** | 起跳动画（播一次） | 动画快播完，或角色已经离地 | 时间短且固定，放 SM 里就够 |
| **InAir** | 下落循环动画（一直播） | 角色落地了（`bIsFalling == false`） | **时间不固定**，必须循环等待 |
| **Land** | 落地恢复动画（播一次） | 动画播完 | 时间短且固定，放 SM 里就够 |

---

## 那什么动作用 Montage，什么用 State Machine？

判断标准就一条：

> **这个动作的持续时间是固定的，还是需要等某个条件才结束？**

| 情况 | 用什么 | 例子 |
|---|---|---|
| **固定时间，播完就完** | 都行，但 Montage 更方便管理 | 攻击、翻滚、喝药、开门、受击 |
| **时间不固定，要等条件** | 必须 State Machine | 空中循环、梯子攀爬循环、举盾持续中 |
| **持续存在，根据参数变化** | State Machine + BlendSpace | 走路、跑步、锁定移动 |

---

## 你说的"不是无级变化所以用 Montage"这个判断对不对？

**部分对，但不完全准确。**

更准确的说法是：

| 你的判断 | 修正 |
|---|---|
| "无级变化 → BlendSpace" | ✅ 正确 |
| "不是无级变化 → Montage" | ❌ 不完全对 |

不是无级变化的动画，有两种情况：

```
情况1：播一次就完，时间固定
  → Montage ✅
  → 例如：攻击、翻滚、喝药

情况2：不是无级变化，但持续时间不固定
  → State Machine ✅
  → 例如：空中循环（不知道飘多久）、举盾站着（不知道举多久）
```

---

## 所以完整的 State Machine 长这样

## 所以触发下一段的变量是什么？

不是 Bool 组合，是**C++ 代码里的几个简单变量**：

| 变量 | 谁设置的 | 作用 |
|---|---|---|

## 那"不让玩家打断"是怎么做的？

| 你想的方式 | 实际的方式 |

```

  挥第一刀 → 在挥的过程中按第二下 → 被吞了 → 第一刀结束 → 没反应
  → 玩家："我明明按了！垃圾游戏！"

如果用输入缓存：
  挥第一刀 → 在挥的过程中按第二下 → 存起来 → 窗口开了 → 自动执行
  → 玩家："手感真好，行云流水！"
```

所以流程是：

```
┌─────────────────────────────────────────────────────┐
│ 玩家随时可以按键，代码随时接受输入                      │
│                                                      │
│ 但是：                                                │
│                                                      │
│ 当前在做动作（bIsPerformingAction == true）            │
│   ├─ 连段窗口开着（bCanChain == true）                │
│   │   └─ 立刻执行下一段                               │
│   │                                                   │
│   └─ 连段窗口没开                                     │
│       └─ 存到缓存里，等窗口开了再执行                   │
│          缓存有保质期（0.2秒），过期就扔掉              │
│                                                      │
│ 当前没在做动作（bIsPerformingAction == false）          │
│   └─ 直接执行                                         │
└─────────────────────────────────────────────────────┘
```

|---|---|---|---|
| `ANS_ComboWindow` | 同类型普通连段 | 较晚（0.5） | 必须让当前招式打出实质效果后才能接 |
