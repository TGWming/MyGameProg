## 第一章 · 核心设计原则（必须固定，不可动摇）

```text
================================================================================
  1.1 八条核心原则
================================================================================

以下八条原则经三方验证一致同意，是整个系统的根基。
任何后续开发决策如果与这八条冲突，必须回到这里重新审视。

原则1：动作请求制（TryDoAction）
- 所有动作执行必须通过统一的请求接口
- 不允许任何地方直接调用 PlayMontage
- 请求流程：输入 → 条件检查 → 状态验证 → 执行 → 回调
- 玩家、敌人、Boss 共用同一套请求流程
- 只是触发来源不同（Input vs BehaviorTree）

原则2：C++ 管规则，AnimBP 管表现
- C++ / Character / Component 负责"为什么播这个动画"
- AnimBP 负责"怎么播这个动画"
- AnimBP 中不包含任何战斗判定、耐力计算、打断逻辑
- AnimBP 只读取状态变量并据此选择动画

原则3：玩家角色不使用 Blackboard 控制动画状态
- Blackboard 专门给 AI 决策使用（配合 BehaviorTree + AIController）
- 玩家角色的动画状态由 Character / Component 中的枚举和变量驱动
- AnimBP 不关心"谁在控制角色"，只关心"当前状态是什么"
- 如果未来有 AI 接管玩家的需求：
  AI 修改 Character 状态 → AnimBP 读取 → 播放动画
  与玩家操控时完全相同的后半段路径

原则4：State Machine 处理持续状态，Montage 处理离散动作
- State Machine 适合：Idle/Move/Fall/Land/Ladder/Guard Loop 等持续性状态
- Montage 适合：攻击/闪避/受击/喝药/交互/处决/技能等离散事件
- 原因：类魂项目的离散动作需要精准时机控制
  （命中窗口、无敌帧、霸体帧、连段窗口、输入缓存）
  这些通过 Montage + Notify 管理远优于纯 State Machine

原则5：玩家 / 敌人 / Boss 统一状态设计语言
- 所有角色类型共用同一套状态枚举定义
- 所有角色类型共用同一套动作请求接口思想
- AnimBP 可以不同，但变量接口命名保持一致
- 这确保了：加新敌人不需要发明新状态系统

原则6：Root Motion From Montages Only
- AnimBP 的 Root Motion Mode 设为 Root Motion From Montages Only
- Blend Space 中的移动动画使用 In-Place 动画
  由 CharacterMovementComponent 控制移动
- 只有 Montage 播放时才使用 Root Motion
  （闪避、突进攻击、处决等需要动画驱动位移的动作）
- 这确保移动系统和 Root Motion 不会互相抢控制权

原则7：数据驱动动作配置
- 动作数据通过 DataTable / DataAsset 配置
- 不把动作规则写死在 AnimBP 或散落的蓝图中
- 包括：Montage引用、耐力消耗、窗口时间、连段关系、AI使用参数等
- 这确保了：换武器、加动作、调参数不需要改代码

原则8：先完成最小战斗闭环，再扩展
- 不要试图一次做完所有动画和系统
- 先完成玩家的最小可战斗闭环
- 验证手感和架构后，再扩展到敌人和 Boss
- 闭环内容见本文档第七章

================================================================================
  1.2 UE4.27 平台设计原则（12条）
================================================================================

1.  优先用成熟方案，不追求 UE5 风格复杂技术
2.  动画核心以 AnimBP + State Machine + Montage 为主
3.  DataTable/DataAsset 用于动作配置，不把规则写死在 AnimBP
4.  AnimBP 只做播放逻辑，不做复杂战斗判定
5.  Character 或 Component 管状态，AnimBP 管表现
6.  AI 用 Blackboard/BT 决策，不直接替代动画状态机
7.  Root Motion 使用要统一策略，不能一半依赖一半不用
8.  玩家与敌人的状态命名和接口尽量统一
9.  能复用的通知、动作数据结构、组件接口都要抽象
10. 先建立最小闭环，再扩展到 Boss 与特殊敌人
11. 先可维护，再追求复杂表现
12. Debug 可视化必须同步建设，否则后期必乱

================================================================================
  1.3 层级优先级模型
================================================================================

整个角色动画系统的概念层级（从高到低）：

  ┌─────────────────────────────────────┐
  │  State/Mode Layer（模式层）          │  ← 最高层
  │  决定角色当前处于什么大状态          │
  │  Normal / Combat / Climb / Dead     │
  ├─────────────────────────────────────┤
  │  Locomotion Layer（移动层）          │  ← 受 Mode 影响
  │  不同 Mode 下移动方式不同            │
  │  Idle / Move / Sprint / Strafe      │
  ├─────────────────────────────────────┤
  │  Action Layer（动作层）              │  ← 覆盖在移动之上
  │  离散动作，可打断/覆盖移动           │
  │  Attack / Dodge / Heal / Interact   │
  ├─────────────────────────────────────┤
  │  Reaction Layer（反应层）            │  ← 最高中断优先级
  │  可打断 Action 和 Locomotion         │
  │  HitReact / Stagger / Death         │
  └─────────────────────────────────────┘

注意：这是概念优先级图，不是代码嵌套结构。
代码中采用多枚举并行 + 优先级门控的方式实现。
不要因为概念图是层级的，就在代码中做成嵌套 if-else 树。
```

## 第二章 · 统一状态系统设计

```text
================================================================================
  2.1 状态拆分原则
================================================================================

绝对不要做的事：
把所有状态塞进一个巨型枚举：
  Idle / Run / Attack / Hit / Dead / CombatIdle / CombatRun /
  CombatAttack / CombatHit / LockOnRun / LockOnAttack ...
这会导致状态爆炸，不可维护。

正确做法：
拆成多个独立枚举，AnimBP 根据多个变量组合决定播放什么动画。

例如：
  MainState       = Combat
  LocomotionState = Move
  OverlayState    = OneHandedWeapon
  MovementStyle   = LockOnStrafe
  Gait            = Run
  ActionState     = None

AnimBP 读取这些变量的组合，选择对应的 Blend Space 和状态。

================================================================================
  2.2 统一状态枚举定义（全角色通用）
================================================================================

以下枚举适用于所有角色类型（玩家/敌人/Boss/NPC敌对角色）。

--- ECharacterMainState ---
角色当前的顶层状态，决定允许哪些行为。

  Normal          正常探索
  Combat          战斗中
  Interaction     交互中（开门/拉杆/对话）
  Climb           攀爬中
  HitReact        受击硬直中
  Execution       处决/被处决中（双人同步特殊状态）
  Dead            死亡
  Disabled        输入禁用（过场/特殊事件）
  Cinematic       过场动画中

--- ELocomotionState ---
角色当前的运动阶段。

  Idle            静止
  Start           起步
  Move            移动中
  Stop            停步
  Pivot           急停转向
  TurnInPlace     原地转身
  JumpStart       起跳（如果项目保留跳跃）
  InAir           空中
  Land            落地
  LadderMove      梯子移动

注意：Sprint 不在这里，它属于 EGait。
注意：Dodge 不在这里，它属于 EActionState（通过 Montage 执行）。

--- EGait ---
移动速度档位。

  Walk            走
  Run             跑
  Sprint          冲刺

--- EMovementStyle ---
移动风格，决定使用哪个 Blend Space。

  FreeMove        自由移动（非锁定）
  LockOnStrafe    锁定环绕移动
  CautiousWalk    谨慎行走（低姿态/恐怖场景）
  Ladder          梯子移动

--- ERotationMode ---
角色朝向控制方式。

  OrientToMovement    朝向移动方向（非锁定时）
  OrientToTarget      朝向目标（锁定时）
  ManualControl       手动控制（特殊场景）

--- EOverlayState ---
角色当前的装备/姿态叠加层。

  Unarmed             空手
  OneHandedWeapon     单手武器
  TwoHandedWeapon     双手武器
  Firearm             火器
  Torch               火把
  Special             特殊

--- EActionState ---
角色当前正在执行的离散动作。

  None            无动作
  LightAttack     轻攻击
  HeavyAttack     重攻击
  ChargedAttack   蓄力攻击
  Dodge           闪避
  StepBack        后撤步
  Guard           防御
  Parry           弹反
  UseItem         使用道具
  Interact        交互
  Execution       处决执行
  Skill           技能
  Heal            治疗

--- EHitReactState ---
角色当前的受击状态。

  None            无受击
  LightHit        轻受击
  MediumHit       中受击
  HeavyHit        重受击
  Stagger         失衡
  Knockdown       倒地
  GetUp           起身
  Death           死亡

================================================================================
  2.3 辅助状态变量（布尔/浮点，非枚举）
================================================================================

以下变量配合枚举使用，用于更细粒度的状态判断。
在 P0 阶段使用这些变量替代独立的 ECombatSubState 枚举。
如果后期系统复杂度上升，可考虑将部分变量统一为枚举。

  bIsInCombat           bool    是否处于战斗状态
  bIsLockedOn           bool    是否锁定目标
  bIsGuarding           bool    是否正在防御
  bInRecovery           bool    是否在动作恢复期（硬直/收招）
  bCanChain             bool    是否处于可连段窗口
  bCanBeInterrupted     bool    当前动作是否可被打断
  bIsStaggered          bool    是否处于破绽/被弹反状态
  bHasSuperArmor        bool    是否有霸体
  bIsInvincible         bool    是否在无敌帧中
  fCurrentSpeed         float   当前移动速度
  fMovementDirection    float   当前移动方向角度
  fForwardAxis          float   前后轴输入值（锁定时用）
  fRightAxis            float   左右轴输入值（锁定时用）

================================================================================
  2.4 状态系统的通用性说明
================================================================================

以上所有枚举和变量对所有角色类型通用。
不同角色类型的区别不在状态定义，而在：

  玩家：
  - 状态变更由 Input → Character/Component 驱动
  - 不使用 Blackboard
  - 有输入缓存系统

  敌人/Boss：
  - 状态变更由 BehaviorTree → AIController → Character/Component 驱动
  - 使用 Blackboard 做决策（但 Blackboard 不直接控制动画）
  - 不需要输入缓存，但有 AI 冷却和决策逻辑

  两者最终汇聚到同一个路径：
  Character/Component 状态 → AnimBP 读取 → 播放动画
  AnimBP 完全不关心"谁在控制角色"
```

## 第三章 · 动画系统架构

```text
================================================================================
  3.1 通用角色动画架构五层模型
================================================================================

任何角色（玩家/敌人/Boss/NPC敌对角色）都统一具备以下五层：

  一、基础数据层
  - Skeleton（骨骼）
  - Physics Asset（物理资产）
  - Animation Sequences（动画序列）
  - Blend Space（混合空间）
  - Montage（蒙太奇）
  - AnimBP（动画蓝图）
  - DataTable / DataAsset（动作数据配置）

  二、逻辑层
  - Character Base（C++角色基类）
  - Character Components（Combat / Stats / Targeting / Detection）
  - AnimInstance Base（C++可选，推荐）
  - AIController + BehaviorTree + Blackboard（仅AI角色）

  三、状态层
  - MainState
  - LocomotionState
  - ActionState
  - OverlayState
  - HitReactState
  - MovementStyle
  - RotationMode
  - Gait

  四、播放层
  - State Machine 处理持续状态
  - Blend Space 处理连续移动参数
  - Montage 处理离散动作事件
  - Additive 处理上半身/细节修正/呼吸/疲劳
  - Notify / NotifyState 处理时机回调

  五、反馈层
  - VFX（视觉特效）
  - SFX（音效）
  - Camera Shake（相机震动）
  - Hit Pause（命中顿帧）
  - UI 提示
  - Debug 可视化

================================================================================
  3.2 Blend Space 使用原则
================================================================================

适合做 Blend Space 的（连续变化、参数驱动）：
  - 普通移动（Speed + Direction）
  - 锁定环绕移动（Forward/Right 或 Speed + Direction）
  - 战斗待机移动
  - 受伤移动（如果有）
  - 爬梯移动
  - 警戒巡逻移动（敌人）

不适合做 Blend Space 的（离散事件、高优先级行为）：
  - 攻击连段
  - 处决 / 背刺
  - 喝药
  - 拉杆 / 开门
  - 受击大硬直
  - 倒地 / 起身
  - 死亡
  - 特殊技能
  - Boss 转阶段

Lock-on Strafe Blend Space 特别说明：
  - 类魂项目最重要的 Blend Space 之一
  - 推荐 2D Blend Space：
    X轴：Right（-1 到 1）
    Y轴：Forward（-1 到 1）
  - 至少 9 个采样点（8方向 + 中心 Idle）
  - 如需区分 Walk/Run，考虑额外 Speed 轴或分两个 BS

================================================================================
  3.3 AnimBP 结构模板
================================================================================

所有角色的 AnimBP 遵循同一套组织结构模板。
不是共用一个 AnimBP，而是共用结构模板。

  1. Event Graph
     - 从 Character / Components 拉取状态
     - 缓存速度、方向、是否锁定、是否战斗、是否受击等
     - 统一处理 Notify 回调

  2. AnimGraph
     - Base Locomotion State Machine
     - Combat Locomotion State Machine（或合并到 Base 内通过参数切换）
     - Overlay / Additive Layer
     - Slot（Montage）
     - Aim/Look Offset（如果需要）
     - Layered Blend Per Bone（上半身覆盖，类魂项目中谨慎使用）
     - Cached Pose（节制使用，不超过 3-4 个）
     - Turn In Place

  3. State Machine 分层
     - Grounded（推荐半嵌套结构）
     - InAir
     - Locomotion
     - HitReact（作为门控状态，阻止 Locomotion 输出）
     - Ladder（如有）
     - Dead

  4. Montage Slot 规划
     - DefaultGroup.FullBody     全身动作（攻击/闪避/处决）
     - DefaultGroup.UpperBody    上半身覆盖（喝药/换武器/投掷）
     - DefaultGroup.Additive     Additive层（呼吸/疲劳/受伤）
     
     注意：Slot Group 必须在 Skeleton 的 Anim Slot Manager 中提前配置
     如果不预先规划，后期改动会影响所有已有 Montage

  Layered Blend Per Bone 在类魂项目中的使用边界：
  - 适合：喝药时下半身可移动、持火把上半身叠加、轻微 Additive 表现
  - 不适合：攻击动作的上下半身分离（会失去类魂攻击的重量感）

================================================================================
  3.4 State Machine 最小集合
================================================================================

--- 玩家角色 ---

  Grounded SM（半嵌套结构）：
  ├── Idle State
  │   └── 根据 MainState 选择 Exploration Idle 或 Combat Idle
  ├── Movement State
  │   └── 根据 MovementStyle 选择对应 Blend Space
  │       ├── FreeMove BS
  │       ├── LockOnStrafe BS
  │       └── Gait 参数控制速度档
  ├── Turn In Place State
  └── Start/Stop State（可后期加入）

  InAir SM：
  ├── Fall
  └── Land

  Ladder SM（如有）：
  ├── Enter
  ├── Loop
  └── Exit

  Dead SM：
  └── Death Pose / Loop

  攻击/闪避/受击 → 不在 SM 中，走 Montage Slot
  HitReact → SM 中保留门控状态（不播放动画，只阻止 Locomotion 输出）
              实际受击动画由 Montage 处理
              Montage 结束后回到 Grounded

--- 敌人/Boss ---

  Grounded SM：
  ├── Idle
  ├── Patrol Move
  ├── Alert Move
  ├── Combat Move
  └── Strafe Move（精英/Boss）

  InAir SM（若需要）：
  ├── Fall
  └── Land

  HitReact SM（门控，同玩家处理方式）：
  └── 阻止 Locomotion → Montage 播放受击 → 结束后返回 Grounded

  Special State（Boss）：
  ├── Phase Change
  ├── Enraged
  ├── Summon
  └── Cast

  Dead SM

================================================================================
  3.5 Montage 与 State Machine 的职责对照
================================================================================

  State Machine 承载：                Montage 承载：
  ─────────────────                  ─────────────────
  Idle / Move / LockOnMove           轻攻击 / 重攻击
  Combat Idle / Combat Move          闪避 / 垫步
  InAir / Land                       喝药 / 使用道具
  Ladder                             交互（开门/拉杆）
  Guard Loop（视需求）                受击（尤其需要窗口控制时）
  Downed / GetUp（也可混合）          处决 / 背刺
                                     技能
                                     武器切换
```

## 第四章 · 动作请求系统设计

```text
================================================================================
  4.1 动作请求制 vs 直接播放制
================================================================================

绝对不要这样做：
  按下攻击键 → 直接在蓝图里 Play Montage Attack_01

必须这样做：
  按下攻击键
  → Character/C++ 发起 TryDoAction(EActionState::LightAttack)
  → 检查当前 MainState 是否允许
  → 检查当前 ActionState 是否允许（不在不可打断的动作中）
  → 检查耐力是否足够
  → 检查是否在可连段窗口（决定是普攻1还是普攻2）
  → 从 DataTable 查找对应的 ActionData
  → 设置 ActionState
  → 扣除耐力
  → 通知 AnimInstance / 播放对应 Montage
  → Montage Notify 回传命中窗口 / 连段窗口 / 结束事件
  → Character 更新状态

优点：
  - 玩家、敌人、Boss 统一
  - 后期加武器不崩
  - 后期加敌人不崩
  - 易于调试
  - 易于复用

================================================================================
  4.2 完整动作请求流水线（以玩家轻攻击为例）
================================================================================

  步骤1：Input（蓝图/C++）
  → 玩家按下攻击键

  步骤2：InputBuffer（C++）
  → 如果当前不能立即攻击，缓存这个输入
  → 缓存有效期约 0.15~0.3 秒（可调参）
  → 当进入"可接受输入"窗口时自动触发

  步骤3：TryDoAction（C++，CombatComponent）
  → 检查 MainState 是否允许（不在 Dead/Disabled/Cinematic）
  → 检查 ActionState 是否允许（不在不可打断的动作中）
  → 检查 Stamina 是否足够
  → 检查 ComboWindow 是否开启（决定连段序号）
  → 从 DataTable 查找对应的 ActionData

  步骤4：ExecuteAction（C++，CombatComponent/ActionComponent）
  → 设置 ActionState = LightAttack
  → 扣除 Stamina
  → 调用 AnimInstance->Montage_Play(ActionData.Montage)
  → 如果需要锁定朝向：修改旋转模式
  → 如果使用 Root Motion：确认设置正确
  → 触发 CameraEvent 回调（预留相机系统联动接口）

  步骤5：Notify 回调（AnimBP → C++）
  → ANS_ComboWindow：开启/关闭连段窗口
  → AN_EnableHitbox：激活攻击碰撞
  → AN_DisableHitbox：关闭攻击碰撞
  → AN_PlaySFX：触发挥砍音效
  → AN_PlayVFX：触发挥砍特效
  → AN_ActionEnd：动作结束，重置 ActionState

  步骤6：中断处理（C++）
  → 如果在攻击过程中被击中
  → HitReact 逻辑判断是否应该被打断（韧性/霸体）
  → 如果打断：停止当前 Montage，播放受击 Montage
  → 重置 ActionState

  对敌人同样适用：
  → 敌人的 BT Task 调用同一个 TryDoAction
  → 触发来源从 Input 变成 BT Decision
  → 后续流程完全一致

================================================================================
  4.3 输入缓存系统
================================================================================

类魂项目的输入缓存是手感的核心之一。

  原理：
  玩家在动作执行中按下的输入不会丢失
  而是被缓存一小段时间
  当角色进入"可接受输入"窗口时
  缓存的输入自动触发

  关键参数：
  - BufferDuration：缓存有效时长（0.15~0.3秒）
  - BufferAction：缓存的动作类型
  - bIsBufferActive：缓存是否激活

  这个系统必须用 C++ 实现，蓝图不适合处理这种帧精度逻辑。

================================================================================
  4.4 连段系统
================================================================================

  连段不是"按一次打一下"
  而是通过 Notify 控制的窗口系统：

  Montage 播放中：
  ├── 攻击开始
  ├── [ANS_ComboWindow 开启]  ← 此窗口内的输入会触发下一段
  ├── [AN_EnableHitbox]       ← 攻击判定开始
  ├── [AN_DisableHitbox]      ← 攻击判定结束
  ├── [ANS_ComboWindow 关闭]  ← 窗口关闭
  └── 恢复期 → ActionEnd

  如果在 ComboWindow 内接收到输入：
  → Montage_JumpToSection("Combo_02")
  → 或 Montage_Play(NextComboMontage)

  如果 ComboWindow 关闭仍无输入：
  → 播放完恢复期 → 回到 Idle/Move
```

## 第六章 · Root Motion 策略

```text
================================================================================
  6.1 Root Motion 前置决策（必须在导入动画资产前确定）
================================================================================

  这个决策必须在阶段1.5完成，在导入任何动画资产之前。
  如果动画资产有 Root Motion 数据但你不想用
  或者你想用但资产里没有
  后期返工代价很大。

  决策表：

  ┌──────────────────┬──────────────┬────────────────────────────────┐
  │ 动画类型          │ 是否用 RM    │ 原因                           │
  ├──────────────────┼──────────────┼────────────────────────────────┤
  │ 移动类动画        │ 不使用       │ 由 CharacterMovement 控制       │
  │ （Walk/Run/Idle） │              │ BS 中应使用 In-Place 动画       │
  ├──────────────────┼──────────────┼────────────────────────────────┤
  │ 攻击动画          │ 视设计需求   │ 需要位移的攻击（突进斩）→ 用   │
  │                   │              │ 不需要位移的 → 不用             │
  │                   │              │ 建议：默认不用，需要时单独开启  │
  ├──────────────────┼──────────────┼────────────────────────────────┤
  │ 闪避动画          │ 强烈建议使用 │ 闪避距离和节奏由动画决定更自然  │
  │                   │              │ 类魂闪避手感很大程度取决于 RM   │
  ├──────────────────┼──────────────┼────────────────────────────────┤
  │ 受击动画          │ 一般不使用   │ 受击位移由代码控制更灵活         │
  │                   │              │ （Launch / Knockback 力）       │
  ├──────────────────┼──────────────┼────────────────────────────────┤
  │ 处决动画          │ 使用         │ 双人同步需要精确位移             │
  └──────────────────┴──────────────┴────────────────────────────────┘

================================================================================
  6.2 UE4.27 Root Motion 设置
================================================================================

  AnimBP 设置：
  - Root Motion Mode = Root Motion From Montages Only
  - 这样 Blend Space 里的动画不受 Root Motion 影响
  - 只有 Montage 播放时才使用 Root Motion

  Montage 设置：
  - 每个 Montage 可单独开关 EnableRootMotion
  - 需要位移的动作开启，不需要的关闭

================================================================================
  6.3 UE4.27 特有限制
================================================================================

  Motion Warping：
  - UE4.27 没有原生 Motion Warping
  - 如果需要攻击自动修正朝向/距离（类魂常见需求）
  - 需要自己实现简化版：
    a. 攻击开始时计算目标位置
    b. 通过代码插值修正角色朝向
    c. 或通过 Root Motion Scale 修正位移距离
  - 建议放在 P1 阶段实现

  Distance Matching：
  - UE4.27 没有 UE5 的 Distance Matching 节点
  - Start/Stop 动画的脚步匹配需要手动处理
  - 建议前期直接用混合过渡，不在此花费太多时间
```

## 第八章 · AI 系统与 Blackboard

```text
================================================================================
  8.1 Blackboard 的定位
================================================================================

  重申：Blackboard 是给 AI 决策用的，不是给动画状态机用的。

  Blackboard 不直接控制"播哪一帧动画"
  而是控制：
  - 当前是否进入战斗
  - 是否接近目标
  - 是否使用技能
  - 是否需要撤退
  - 是否转阶段

  然后：
  AIController / Character / Component
  → 修改角色状态（统一枚举）
  → AnimBP 读取状态
  → State Machine / Montage 播放对应动作

================================================================================
  8.2 敌人/Boss Blackboard 推荐字段
================================================================================

  --- 核心战斗字段 ---
  TargetActor                   当前目标
  DistanceToTarget              到目标距离
  HasLineOfSight                是否有视线
  IsAlerted                     是否警觉
  IsInCombat                    是否战斗中
  CanAttack                     是否可攻击
  CanStrafe                     是否可环绕

  IsStunned                     是否被击晕
  IsDead                        是否死亡
  DesiredRange                  期望战斗距离
  CurrentPhase                  当前阶段（Boss）
  IsPhase2                      是否二阶段（Boss）

  --- 导航与巡逻字段 ---
  HomeLocation                  出生位置
  SpawnLocation                 刷新位置

  PatrolIndex                   巡逻点索引
```

## 第十章 · Notify 与音效联动规范

```text
================================================================================
  10.1 Notify 命名规范
================================================================================

  瞬时通知（单一时间点）使用 AN_ 前缀：
  - AN_PlaySFX_SwingLight         挥砍音效（轻）
  - AN_PlaySFX_SwingHeavy         挥砍音效（重）
  - AN_PlaySFX_FootStep           脚步音效
  - AN_PlayVFX_Slash               斩击特效
  - AN_PlayVFX_Impact              命中特效
  - AN_EnableHitbox                攻击判定开启
  - AN_DisableHitbox               攻击判定关闭
  - AN_ActionEnd                   动作结束

  状态通知（持续时间段）使用 ANS_ 前缀：
  - ANS_ComboWindow                连段窗口
  - ANS_SuperArmor                 霸体窗口
  - ANS_IFrame                     无敌帧窗口
  - ANS_DisableInput               禁用输入窗口
  - ANS_EnableHitbox               持续攻击判定（可替代 AN 版本）

  所有 Notify 建议用 C++ 定义（UAnimNotify/UAnimNotifyState 子类）
  在 AnimBP 中添加使用

================================================================================
  10.2 音效联动原则
================================================================================

  战斗音效时机要求精确：
  - 挥刀音效：必须和动画的挥砍帧精确对齐（通过 AN_PlaySFX）
  - 命中音效：由命中判定触发（不是 Notify，是碰撞回调）
  - 脚步音效：必须和落脚帧对齐（通过 AN_PlaySFX_FootStep）
  - 防御/弹反音效：由判定窗口触发

  在 Notify 中只发出事件
  实际音效播放由音效系统（蓝图侧）负责
  保持"动作系统触发事件，表现系统响应事件"的原则

================================================================================
  10.3 相机联动预留
================================================================================

  在 ExecuteAction 中预留 CameraEvent 回调接口
  让相机组件可以订阅动作事件并做出响应

  需要响应的动作事件：
  - 攻击时的轻微 Camera Shake
  - 受击时的相机反馈
  - 处决时的特殊相机角度
  - 闪避时相机跟随/延迟
  - Boss战的相机距离自动调整

  接口预留应在 P0 阶段完成（你已有相机组件，联动接口越早做越好）
```
