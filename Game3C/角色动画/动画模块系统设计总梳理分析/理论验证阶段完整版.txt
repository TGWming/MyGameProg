# 一、先给结论：你现在最需要的不是“先做更多动作”，而是先统一“动作架构”

你现在的问题，本质上不是缺某个动画，而是缺一套**统一角色动作管线**。  
对于你这个项目（UE4.27、类魂、C++/蓝图混合开发），推荐采用这套核心原则：

```text
1. 不要把“每个动作”都直接写成分散逻辑
2. 先建立统一的“角色动作分层架构”
3. 玩家、敌人、Boss 共用同一套设计语言，但允许资产和细节不同
4. 动画驱动表现，代码驱动规则，蓝图驱动拼装
5. Locomotion（移动）/ Action（动作）/ HitReact（受击）/ State（状态）分层
6. AnimBP 负责“怎么播”
7. Character/Component/C++ 负责“为什么播”
8. AI 决策用 Blackboard + Behavior Tree
9. 玩家角色本身一般不依赖 Blackboard 做动作状态机
10. 玩家和敌人都应有统一的 Gameplay State / Action State / Overlay State / Locomotion State
11. 尽量减少“状态爆炸”，避免把每个动作都做成独立 State
12. 优先使用 Montage 承载高优先级动作；State Machine 承载持续性状态
```

# 二、先纠正一个关键认知：UE4.27 里玩家角色动画主线，不建议依赖 Blackboard

这是很重要的一点。

## 1）Blackboard 的主要用途
Blackboard 主要是给 **AI 行为决策** 用的，通常配合：

- Behavior Tree
- AIController
- 感知系统
- 敌人战斗状态切换

也就是说：

- **敌人/BOSS**：很适合用 Blackboard
- **玩家角色**：通常**不需要**依赖 Blackboard 做动画状态控制

## 2）玩家角色动画应该怎么做
玩家角色更推荐：

- **Character / ActorComponent / C++** 维护状态
- **AnimInstance / AnimBP** 读取状态并驱动动画
- **Montage** 处理攻击、闪避、喝药、处决、交互等高优先动作
- **State Machine + Blend Space** 处理持续状态，如移动、战斗移动、受伤硬直、倒地、攀爬等

## 3）你这个项目最合理的分工
```text
玩家：
- 输入 -> Character/C++ -> Gameplay State / Action State
- AnimBP 根据这些状态切换 Locomotion、战斗姿态、回避、攻击、受击等
- 不需要 Blackboard 作为主状态来源

敌人/Boss：
- 感知/距离/仇恨/阶段 -> Blackboard
- AIController / BT 决策“追击、绕圈、攻击、撤步、转阶段”
- Character / Components 执行动作
- AnimBP 同样读取统一状态变量
- Blackboard 不直接等于动画状态机，但可以提供动画决策输入
```

---

```text

- Stagger

第四层：State/Mode Layer（模式层）
- Normal Exploration
```

### 为什么这样拆
因为类魂游戏不是普通动作游戏，它的动作不是单一“播放动画”，而是**模式切换 + 动作覆盖 + 判定窗口 + 资源消耗**的组合。

# 六、是否所有动作都要做 Blend Space？答案：不是

这是第二个关键点。

## 6.1 适合做 Blend Space 的
**连续变化、参数驱动** 的动作适合 Blend Space：

```text
适合 Blend Space：
- 普通移动（Speed + Direction）
- 锁定移动（Forward/Right 或 Speed + Direction）
- 战斗待机移动
- 受伤移动（如果有）
- 爬梯移动
```

```text
不适合 Blend Space：
- 攻击连段
- 处决
- 背刺
- 喝药
- 拉杆开门
- 受击大硬直
- 倒地
- 起身
- 死亡
- 特殊技能
- Boss 转阶段
```

# 七、推荐的 AnimBP 结构：UE4.27 类魂项目标准模板

我建议你每个角色类型都遵循同一套 AnimBP 组织结构。  
不是完全共用一个 AnimBP，而是**共用结构模板**。

---

## 7.1 AnimBP 建议分区

```text
AnimBP 建议分为：

1. Event Graph
- 从 Character / Components 拉取状态
- 缓存速度、方向、是否锁定、是否在空中、是否战斗中、是否受击等

2. AnimGraph
- Base Locomotion State Machine
- Combat Locomotion State Machine
- Overlay / Additive Layer
- Slot（Montage）
- Aim/Look Offset（如果需要）
- IK / Foot Placement（后期）
- Turn In Place
- Layered Blend Per Bone（上半身覆盖）
- Cached Pose

3. State Machine 分层
- Grounded
- InAir
- Locomotion
- Combat Locomotion
- HitReact
- Ladder
- Dead

4. Montage Slot 规划
- FullBody
- UpperBody
- HitReact
- Additive
- Execution/Cinematic
```

---

## 7.2 一个非常重要的原则：State Machine 承载“持续状态”，Montage 承载“动作事件”

```text
State Machine 适合：
- Idle/Move/LockOnMove
- Combat Idle / Combat Move
- InAir / Land
- Ladder
- Downed / GetUp（也可混合）
- Guard Loop（看需求）

Montage 适合：
- 轻重攻击
- 闪避
- 喝药
- 交互
- 受击（尤其需要窗口控制时）
- 处决
- 技能
- 武器切换
```

### 对你这个项目的建议
类魂项目里，**攻击、闪避、受击、交互** 大多建议用 **Montage**。  
因为这些动作通常需要：

- 精准时机
- 根运动 / 位移控制
- 命中窗口
- 霸体窗口
- 可打断窗口
- 输入缓存
- 连段窗口

这些都比纯 State Machine 更适合 Montage 管理。

# 十一、C++ 和蓝图到底怎么分工最合理

你问得很关键：  
**最后是否需要用 C++，或者单纯蓝图来变换玩家角色不同动作？**

我的结论是：

```text
结论：
1. 可以纯蓝图做，但你的项目不建议
2. 你的项目类型���适合“C++管理规则 + 蓝图/AnimBP管理表现”
3. 动画切换本身不一定必须写C++
4. 但动作状态、输入缓存、战斗规则、组件化接口，强烈建议用C++
```

## 11.1 推荐分工表


# 十二、推荐的“动作请求制”而不是“直接播放制”

你后续一定要采用这个思想。

不要这���：

```text
按下攻击键 -> 直接在蓝图里 Play Montage Attack_01
```

应该这样：

```text
按下攻击键
-> Character/C++ 发起 TryDoAction(EActionState::LightAttack)
-> 检查当前状态是否允许
-> 检查耐力是否足够
-> 检查是否在可连段窗口
-> 选择正确的动作数据
-> 通知 AnimInstance / 播放对应 Montage
-> Montage Notify 回传命中窗口 / 连段窗口 / 结束事件
-> Character 更新状态
```

### 优点
- 玩家、敌人、Boss 都能统一
- 后期加武器不崩
- 后期加敌人不崩
- 易于调试
- 易于复用

## 第二节 "Blackboard不用于玩家" — ✅ 完全对齐，补充一点

```text
验证结论：核心观点完全正确

Blackboard 确实是给 AI 决策用的（BehaviorTree + AIController）
玩家角色确实不应依赖 Blackboard 来控制动画状态

⚠️ 补充一点：
在 UE4.27 中，如果你的玩家角色后续可能有"AI接管"场景
（例如：玩家被控制、自动战斗、NPC同伴切换为可控角色），
那么建议在 Character 基类中预留一个 AIController 切换接口，
但动画状态系统本身仍然走统一状态枚举，而不是让 Blackboard 直接驱动动画。

也就是说：
- AI接管时：BT修改Character状态 → AnimBP读取状态 → 播放动画
```

## 第十二节 "动作请求制" — ✅ 完全对齐，这是全纲领最关键的设计

```text
验证结论：完全正确，这一节是整套纲领的灵魂

"动作请求制"（TryDoAction）vs "直接播放制"（直接PlayMontage）
这个区分对于类魂项目来说是决定性的

进一步展开这个系统的推荐实现流程：

完整的动作请求流水线（以玩家轻攻击为例）：

1. Input（蓝图/C++）
   → 玩家按下攻击键

2. InputBuffer（C++）
   → 如果当前不能立即攻击，缓存这个输入
   → 缓存有效期约 0.15~0.3 秒（可调参）
   → 当进入"可接受输入"窗口时自动触发

3. TryDoAction（C++，CombatComponent）
   → 检查 MainState 是否允许（不在 Dead/Disabled/Cinematic）
   → 检查 ActionState 是否允许（不在不可打断的动作中）
   → 检查 Stamina 是否足够
   → 检查 ComboWindow 是否开启（决定是普攻1还是普攻2）
   → 从 DataTable 查找对应的 ActionData

4. ExecuteAction（C++，ActionComponent）
   → 设置 ActionState = LightAttack
   → 扣除 Stamina
   → 调用 AnimInstance->Montage_Play(ActionData.Montage)
   → 如果需要锁定朝向：修改旋转模式
   → 如果使用 Root Motion：确认设置正确

5. Notify 回调（AnimBP → C++）
   → ANS_ComboWindow：开启/关闭连段窗口
   → AN_EnableHitbox：激活攻击碰撞
   → AN_DisableHitbox：关闭攻击碰撞
   → AN_ActionEnd：动作结束，重置 ActionState

6. 中断处理（C++）
   → 如果在攻击过程中被击中
   → HitReactComponent 判断是否应该被打断
   → 如果打断：停止当前 Montage，播放受击 Montage
   → 重置 ActionState

这个流水线对敌人同样适用：
- 敌人的 BT Task 调用同一个 TryDoAction
- 只是触发来源从 Input 变成了 BT Decision
- 后续流程完全一致
```

## 3）第三节：四层模型里，Mode 是否应���放最顶层 —— 合理，但属于“表达优化”，不是原则纠错

这是一个**合理建议**，但我要强调：它更像是**层级表达优化**，而不是“原来错了”。

```text
二次验证结论：
```

为什么我这么说：

### 它对的地方
因为在决策优先级上，确实通常是：


例如：

- Normal 模式下不能用某些战斗动作
- Climb 模式下 Locomotion 逻辑完全不同

所以把 `State/Mode Layer` 放最顶层，**更符合“控制关系”**。

### 但要注意的地方
“层级高低”是**概念图上的优先级**，不一定是你在代码中真的按一层包一层实现。  
项目里更实际的是：

```text
MainState / LocomotionState / ActionState / HitReactState
并行存在，由优先级仲裁器或状态规则共同约束
```

所以我的建议是：

```text
采纳这个表达方式：
- 在理论图和文档图中，把 Mode/State 放最高层

但实现上不要强行做成“只有顶层决定底层”的死板树结构，
仍然保留多枚举组合 + 优先级门控的实现思路。
```

### 4.1 关于 `DodgeMove` 不该放在 `ELocomotionState` —— 我认同，建议采纳

```text
二次验证结论：
- 合理
- 建议采纳
```

原因很简单：

- Dodge 是离散高优先动作
- 它通常通过 Montage 承载
- 它有输入窗口、无敌帧、打断逻辑、耐力消耗
- 这更像 `ActionState`，而不是持续移动状态

所以这条修正是对的。

### 4.2 关于 `Sprint` 更适合放进 `EGait` 而不是 `ELocomotionState` —— 基本认同，建议采纳

```text
二次验证结论：
- 合理
- 推荐采纳
```

因为 Sprint 更多是：

- 移动速度档位
- 移动风格的一种参数分支

而不是一个和 `Idle/Move/Start/Stop` 同级的“运动阶段”。

所以更好的拆法是：

```text
ELocomotionState:
- Idle
- Start
- Move
- Stop
- Pivot
- TurnInPlace
- JumpStart
- InAir
- Land
- LadderMove

EGait:
- Walk
- Run
- Sprint
```

这会让 Blend Space 和状态判断更清晰。

原因：

- 它具有特殊相机、位移对位、无打断/特殊打断规则
- 受击者与执行者都进入一个非常特殊的流程

### 4.4 关于新增 `ECombatSubState` —— 合理，但“是否单独枚举”取决于项目规模

```text
- 建议保留为“可选扩展”
```

为什么？

因为它想解决的问题是真实存在的：

```text
Attacking
Guarding
Dodging
Recovering
Parrying
Staggered
```

这些确实决定当前角色“还能不能做别的动作”“能不能被惩罚”“能不能取消”。

但实现方式不一定非要是独立枚举。你有三种做法：

```text
做法A：单独枚举 ECombatSubState
优点：语义清晰
缺点：状态组合数量增加

做法B：由 EActionState + 若干布尔值/窗口状态组合得到
例如：
ActionState = LightAttack
bInRecovery = true
bCanChain = false

做法C：由动作数据 + 当前窗口动态推导
适合更数据驱动的系统
```

对你现在这个项目阶段，我更建议：

```text
先不强制新增 ECombatSubState
先用：
- ActionState
- bInRecovery
- bIsGuarding
- bCanParry
- bIsStaggered
等状态字段解决问题

等系统复杂到明显需要抽象时，再统一成 CombatSubState
```

```text

当敌人在移动状态中被击中时
如果 HitReact 完全通过 Montage 处理（FullBody Slot）
Montage 会覆盖 Locomotion State Machine 的输出

这本身是期望行为：被击中时停止移动动画，播放受击动画

但问题出在 Montage 结束后的回到移动状态的过渡：
- Montage 结束 → Slot 输出为空 → 回到 SM 输出
- 如果 SM 当前仍在 Move 状态但角色实际已停下
- 会出现一帧的"滑步"或"动画闪烁"

解决方案：
1. HitReact Montage 结束时，通过 Notify 通知 Character
2. Character 重新评估当前状态并更新 SM 输入变量
3. 或者在 Montage 末尾加一个短 BlendOut 到 Idle
4. 或者在 SM 中确实保留一个 HitReact State 作为门控
   这个 State 不播放动画，只是阻止 Locomotion 输出
   等 Montage 结束后再过渡回正确的 Locomotion State

所以二次验证的结论没错，但需要注意这个过渡细节。
这也是为什么原纲领给敌人保留了 HitReact SM 的原因。

修正后的建议：
- 受击动画本身 → Montage 处理 ✅
- SM 中保留一个 HitReact 门控状态 → 推荐保留 ✅
- 这个门控状态不播放动画，只是阻止 Locomotion 输出
- 两者配合使用是最稳的方案
```

## 四、三方纲领合并后仍存在的盲区

二次验证最后提到了两个未覆盖的领域，我对此进行补充：

```text
盲区1：网络同步/联机
二次验证说"你目前看起来是单机项目，可暂不展开"
我的评估：同意，但加一条预防建议

即使当前是单机，在状态系统设计时注意：
- 状态变更集中在 Character/Component 中，不散落在 AnimBP
- 动作请求走统一接口（TryDoAction）
- Montage 播放走统一入口

如果未来有联机需求，这种集中式设计
只需要在请求入口加上 Server/Client 权限判断
而不需要重构整个系统

这不需要现在写网络代码，只需要：
"保持状态变更路径的单一入口"
这一条在单机开发中也是好习惯
```

```text
盲区2：动画重定向与骨骼兼容
二次验证说"如果后续会大量采购/混用外部动作资产，这块最好单独补"
我的评估：正确，而且对于你的项目可能比想象中更早需要

补充建议：

如果你会使用 Mixamo / Marketplace / 其他来源的动画：
1. 尽早确定项目主骨骼（Skeleton）
2. 所有外部动画统一通过 Retarget 到主骨骼
3. 在 UE4.27 中使用 Retarget Manager 设置骨骼映射
4. 建议建立一个 Retarget 验证流程：
   - 导入外部动画
   - Retarget 到主骨骼
   - 在标准场景中预览
   - 检查手脚穿模
   - 检查 Root Motion 是否正确转移
5. 命名规范中加上来源标记：
   AN_Player_LightAttack_01_SRC_Mixamo
   AN_Player_LightAttack_01_SRC_Original
   （或用文件夹区分来源）

这个流程建议在 P0 阶段就确定，避免后期返工。
```

```text
盲区3（新增）：相机系统与动作系统的联动

你已经完成了相机总组件，但纲领中没有详细覆盖
动作系统如何与相机系统配合

类魂项目中相机需要响应的动作事件：
1. 锁定时的相机行为（你的锁定组件已有）
2. 攻击时的轻微相机震动（Camera Shake）
3. 受击时的相机反馈
4. 处决时的特殊相机角度
5. 闪避时相机是否跟随还是延迟
6. Boss战的相机距离自动调整

建议：
在动作请求系统的执行层（ExecuteAction）中
预留一个 CameraEvent 回调接口
让相机组件可以订阅动作事件并做出响应

这不需要现在就全部实现
但接口预留应该在 P0 阶段完成
因为你已经有相机组件了，联动接口越早做越好
```

```text
盲区4（新增）：音效与动作系统的联动规范

你的项目蓝图负责音乐系统和氛围音效
但纲领中没有明确"战斗音效"如何与动作系统对接

对于类魂项目，战斗音效的时机要求非常精确：
- 挥刀音效：必须和动画的挥砍帧精确对齐
- 命中音效：必须和 Hitbox 激活同步
- 脚步音效：必须和移动动画的落脚帧对齐
- 防御/弹反音效：必须和判定窗口同步

推荐方案：
统一通过 AnimNotify 触发
- AN_PlaySFX_SwingLight
- AN_PlaySFX_SwingHeavy
- AN_PlaySFX_FootStep
- AN_PlaySFX_Impact（由命中判定触发，不是Notify）

在 Notify 中只发出事件
实际音效播放由音效系统（蓝图侧）负责
这样保持"动作系统触发事件，表现系统响应事件"的原则

这一点应该在命名规范和Notify规范中补充。
```
