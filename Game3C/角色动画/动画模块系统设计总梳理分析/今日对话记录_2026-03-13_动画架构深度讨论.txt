对话主题 1：两套设计体系的差异分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

背景：
  仓库内存在两套不同架构路线的文档：
  - 文档C（C++类总览与依赖关系完整图.txt）= Montage 驱动路线
  - 最终确认版设计表 = SM 全量 90 State 路线

分析结论：
  两套方案在架构哲学上对立，无法直接合并。
  
  文档C路线：SM只做Locomotion基底，一切动作走Montage+FullBody Slot覆盖
  设计表路线：SM做全量状态承载，7个SM × 90个State全部在SM内管理
  
  最终决策：采用文档C的Montage驱动路线。

对话主题 2：Montage 驱动 vs SM 全量的原理解释
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Montage 驱动：
  State Machine 只管走路/站着/空中/死亡（约6-8个State）。
  攻击、翻滚、喝药、受击等全部由C++代码调用 Montage_Play()，
  通过 FullBody Slot 覆盖在 SM 输出上面。
  比喻：SM是一直在播的电视，Montage是贴在屏幕上的海报（带透明渐变）。

SM 全量：
  State Machine 管一切，90个State，几百条过渡线。
  所有动作切换都在 AnimBP 蓝图里的箭头和条件中完成。

商业案例：
  黑魂3/血源诅咒：自研引擎，用"动作队列+动画覆盖"= Montage等价方案
  匹诺曹（UE4）：直接用 UE Montage 系统
  结论：没有成功的魂like游戏用 SM 全量 90 State 方案。

对话主题 3：Montage Blend 效果讨论
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

用户担忧：Montage 强覆盖会不会 Blend 效果很差？

结论：不会。
  Montage 的 BlendIn/BlendOut 和 SM State 过渡用的是同一套底层混合算法。
  效果完全一致。
  魂like 过渡看起来"快"不是因为 Blend 差，
  而是故意设计成"短BlendIn + 长动画前摇"制造沉重感。
  
  Montage 还能做 SM 做不到的事：
  - BlendIn Curve（自定义混合曲线）
  - 分层 Blend（上下半身分开，如边跑边喝药）
  - Montage 之间的切换也带 Blend

对话主题 4：Locomotion 模块确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

确认要点：
  1. 只需 2 个 BlendSpace（FreeMove + LockOn），不需要更多
  2. BlendSpace 不需要代码"调用"，代码只更新 Speed/Direction 数字
  3. SM 内 Movement State 用 bIsLockedOn 选择哪个 BS
  4. Sprint 用单独 Animation Sequence，不需要 BS
  5. 非锁定移动：角色转身面朝移动方向
  6. 锁定移动：角色面朝敌人，脚步侧移（Strafe 动画）
  7. 锁定 Forward 可复用非锁定 Run_F

  代码职责：每帧算出 Speed/Direction/bShouldMove/bIsFalling 等数字
  SM 职责：根据布尔值决定当前在哪个 State
  BS 职责：根据数字自动混合出具体动画

对话主题 5：跳跃模块确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  跳跃放 State Machine，不用 Montage。
  原因：空中阶段持续时间不固定（跳悬崖飘5秒 vs 跳台阶飘0.3秒）。
  SM 的 InAir State 放循环动画，一直播到落地。
  
  每个 State 里放的是单独的 Animation Sequence，不是 BlendSpace，不是 Montage。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
对话主题 6：翻滚模块确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  翻滚用 Montage，不用 SM。
  虽然翻滚看起来像移动，但特征与攻击一样：
  时间固定、需要 Notify（无敌帧/精力）、Root Motion、可接后续动作。
  
  8方向翻滚 + 后撤步 = 9个 Montage。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
对话主题 7：战斗连段与窗口机制确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  所有攻击都是 Montage。
  连段用 JumpToSection 在同一 Montage 内跳 Section。
  
  Notify 是打开窗口的开关，不是触发 Blend 的东西。
  ANS_ComboWindow 控制 bCanChain。
  窗口内按攻击 → 立刻执行。
  窗口外按攻击 → 输入缓存（0.2秒保质期），窗口开��再执行。
  
  不同优先级动作用不同窗口：
  同一 Montage 上叠多个 NotifyState，位置不同 = 开放时机不同。
  越高优先级的动作窗口越早打开。
  例：翻滚取消窗口最早，变形斩窗口次之，普通连段窗口最晚。
  
  输入缓存不是"禁止输入"，而是"接受输入但缓存等窗口"。
  这是魂like行云流水手感的核心。

对话主题 8：开发顺序确认
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  确认的开发顺序：
  1. 准备所有动画素材
  2. 创建 BlendSpace + 搭建 AnimBP State Machine + 连接 AnimGraph
  3. 创建所有 Montage 并布置 Notify
  4. 以上完成后才开始 C++ 编码（通过 IDE Agent）
