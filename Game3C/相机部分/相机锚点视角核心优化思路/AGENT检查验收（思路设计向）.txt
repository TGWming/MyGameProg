

✅ AGENT 回稿验收 Checklist

（判断：他是不是理解了“构图驱动相机”）

你在看 AGENT 回稿时，逐条对照，不满足就打回。

一、总体理解层（必须全部满足）

☐ 是否明确区分了
构图（Framing）≠ 相机旋转（Rotation）

☐ 是否明确提出一个
Focus Point 作为唯一被看目标

☐ 是否说明 Focus Point 可以不属于任何 Actor

☐ 是否避免使用
“相机看向敌人 / 看向玩家”
这种 Actor 直连表述

👉 任何一条没满足，说明还在旧思维里。

=====================================================================

二、系统分层是否清晰（关键）

AGENT 的方案中，是否明确存在这三层：

☐ 构图意图层（2D 屏幕空间）

☐ Focus 解算层（3D 世界空间）

☐ 相机执行层（Rotation / FOV / Distance）

并且：

☐ 每一层的输入输出是否明确

☐ 是否明确指出
执行层不参与构图决策

如果他把权重、Anchor、Rotation 混在一个模块里，直接否。


=====================================================================

三、Camera State 的理解是否正确

☐ 是否说明 Camera State 定义的是
“构图偏好 + 响应方式”

☐ 是否避免把 Camera State 描述为
“直接决定看向谁”

☐ 是否支持每个 State 拥有
Framing Anchor / Actor Weight / Bias

☐ 是否说明 State 切换 = 构图意图平滑变化

如果 Camera State 只是“参数表”，说明理解不够深。


=====================================================================

四、关于 Boss / 演出相机的解释是否正确

☐ 是否解释为
“构图锚点变化 → Focus Point 移动 → 相机响应”

☐ 是否明确避免
“Boss 状态强制抬头 / 强制改 Rotation”

☐ 是否认为 Boss 不需要特例相机逻辑

如果还在讲“Boss 时直接调 Pitch”，直接打回。

=====================================================================

五、术语是否对齐（非常重要）

☐ Framing Anchor 是否被描述为 2D 屏幕概念

☐ Focus Point 是否被描述为 3D 世界点

☐ Quaternion 是否只作为 执行工具

☐ 是否没有把“四元数”当成区域或锚点

术语错一个，后面一定全错。

=====================================================================

六、整体方案是否具备扩展性

☐ 是否支持未来加入新 Camera State

☐ 是否支持多敌人 / 多目标

☐ 是否不依赖硬编码 Actor 类型

☐ 是否不需要为新 Boss 重写相机逻辑

如果方案一加新状态就要改核心逻辑，直接否。



////////////////////////////////////////////////////////////////////////////////



🚫 执行层不可越界清单

（给你防止“好设计被做坏”用）

这份是红线，任何一条被踩，就要制止。
==========================================================================================================================================

一、执行层（Camera Executor）绝对不能做的事

执行层 不得：

❌ 决定看向哪个 Actor

❌ 参与构图权重计算

❌ 根据 Boss / 技能分支逻辑

❌ 直接改 Framing Anchor 或 Focus Point

它只能：

接收目标值 → 平滑执行

==========================================================================================================================================

二、Rotation 相关的红线

❌ 直接用 LookAt(Actor)

❌ 在 Tick 中直接 SetWorldRotation

❌ 用 Pitch/Yaw 硬修正画面

❌ 因技能触发直接强改 Rotation

Rotation 只能来源于：

相机 → Focus Point 的朝向结果

==========================================================================================================================================

三、Actor 权重相关红线

❌ Actor Weight 直接影响 FOV

❌ Actor Weight 直接影响 Distance

❌ Actor Weight 直接影响 Rotation

权重 只能影响：

Focus Point 的计算

==========================================================================================================================================

四、Camera State 的越界行为

Camera State 不得：

❌ 直接操作相机组件

❌ 在 State 内写 LookAt 逻辑

❌ 绕过 Focus 解算层

Camera State 只能：

提供参数与偏好

==========================================================================================================================================

五、Boss / 演出特例红线（非常常见）

❌ 为 Boss 写专用 Camera Tick

❌ 为某个技能单独控制相机旋转

❌ 用 Timeline 强推相机朝向

正确方式只能是：

改构图意图，让相机“自然反应”

==========================================================================================================================================

六、设计退化的典型信号（看到就警惕）

“这个 Boss 特殊，我们单独处理一下”

“这里直接改 Pitch 会简单很多”

“先用 LookAt 顶一下”

“这个状态不走通用流程”

只要出现一句，系统已经开始退化。
