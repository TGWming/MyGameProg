```cpp
/**
 * ShouldActivate - Determine if auto-orient to movement should be active
 * 
 * R06 activates when: 
 * - Module is enabled
 * - Character is moving (bIsMoving)
 * - Character speed exceeds minimum threshold
 * - No target is locked (free camera mode)
 * - Player is NOT actively controlling camera (no camera input)
 * 
 * This creates natural camera behavior:  when player moves forward without
 * touching camera controls, the camera gradually aligns behind the character. 
 * 
 * Priority 90 (lower than input modules) ensures player input takes precedence.
 * Uses Blend policy with dynamic weight based on movement speed.
 * 
 * Use Cases:
 * - Exploration with automatic camera following
 * - Running forward without manual camera adjustment
 * - Natural camera behavior during navigation
 */
bool UCameraModule_R06_AutoOrient_Movement::ShouldActivate(
	const FStageExecutionContext& Context, 
	const FCameraStateConfig& StateConfig) const
{
	// Check if module is enabled
	if (!bIsEnabled)
	{
		return false;
	}
	
	// Don't activate when target is locked
	// Lock-on modules handle camera in combat
	if (Context.InputContext. bHasTarget)
	{
		return false;
	}
	
	// Check if character is moving
	if (!Context.InputContext.bIsMoving)
	{
		return false;
	}
	
	// Check if character speed exceeds minimum threshold
	// This prevents activation during very slow movement
	float MinSpeed = StateConfig.Module. Rotation.AutoOrientMinSpeed;
	
	// If MinSpeed not configured, use default
	if (MinSpeed <= 0.0f)
	{
		MinSpeed = 100.0f;  // Default:  1 m/s (walking speed)
	}
	
	if (Context.InputContext.CharacterSpeed < MinSpeed)
	{
		return false;  // Too slow, don't auto-orient
	}
	
	// Check if player is NOT actively controlling camera
	// Only auto-orient when player isn't using camera input
	FVector2D CameraInput = Context.InputContext.CameraInput;
	float InputMagnitude = CameraInput.Size();
	
	if (InputMagnitude >= 0.1f)
	{
		return false;  // Player is controlling camera, don't interfere
	}
	
	// All conditions met - activate auto-orient
	return true;
}
```

**说明：**
- 5个条件检查：启用 + 移动 + 速度 + 无锁定 + 无相机输入
- 速度阈值防止极慢移动时激活
- 检测相机输入，玩家控制时不激活
- 优先级90（低于输入模块）

**替换为完整实现：**
```cpp
/**
 * Compute - Calculate rotation to align camera behind moving character
 * 
 * This module calculates the desired camera rotation to position the camera
 * behind the character, facing the direction of movement.  This provides
 * natural camera behavior during exploration and navigation.
 * 
 * Steps:
 * 1. Get character velocity (movement direction)
 * 2. Extract horizontal movement (ignore vertical)
 * 3. Calculate desired camera yaw (facing movement direction)
 * 4. Preserve current pitch (don't affect vertical angle)
 * 5. Calculate dynamic weight based on movement speed
 * 6. Output with blend policy
 * 
 * Key Features:
 * - Camera aligns to movement direction (not character facing)
 * - Only affects yaw (horizontal), preserves pitch
 * - Dynamic weight:  faster movement = stronger effect
 * - Blends with input modules (doesn't override)
 * 
 * @param Context The current pipeline execution context
 * @param StateConfig The current camera state configuration
 * @param OutOutput Output structure to fill with computed values
 * @return True if computation succeeded
 */
bool UCameraModule_R06_AutoOrient_Movement::Compute(
	const FStageExecutionContext& Context, 
	const FCameraStateConfig& StateConfig, 
	FModuleOutput& OutOutput)
{
	// Create base output with module identity
	OutOutput = CreateBaseOutput();
	
	// Step 1: Get character velocity (movement direction)
	FVector Velocity = Context.InputContext.CharacterVelocity;
	
	// Step 2: Extract horizontal movement only
	// We only care about horizontal direction, not vertical (jumping/falling)
	Velocity.Z = 0.0f;
	
	// Validate velocity
	if (Velocity. IsNearlyZero())
	{
		// No horizontal movement, cannot compute orientation
		OutOutput.bHasRotationOutput = false;
		return false;
	}
	
	// Step 3: Calculate desired camera yaw from movement direction
	// Velocity. Rotation() gives the rotation of the movement vector
	FRotator MovementRotation = Velocity.Rotation();
	
	// Step 4: Construct desired camera rotation
	// We want camera to face the movement direction (camera behind character)
	// 
	// Important: We only control yaw, preserve existing pitch
	// This allows player to look up/down while auto-orient handles horizontal
	FRotator DesiredRotation;
	
	// Yaw: Align to movement direction
	DesiredRotation.Yaw = MovementRotation.Yaw;
	
	// Pitch:  Preserve current camera pitch
	// Get current pitch from output or input context
	FRotator CurrentRotation = Context. Output.Rotation;
	if (CurrentRotation.IsNearlyZero())
	{
		// Fallback to previous camera rotation
		CurrentRotation = Context.InputContext.PreviousCameraRotation;
	}
	DesiredRotation.Pitch = CurrentRotation.Pitch;
	
	// Roll: Always zero
	DesiredRotation.Roll = 0.0f;
	
	// Step 5: Calculate dynamic weight based on movement speed
	// Faster movement = stronger auto-orient effect
	// This creates smooth transition and prevents jarring at low speeds
	
	float CharacterSpeed = Context.InputContext. CharacterSpeed;
	
	// Normalize speed to 0-1 range
	// Assume max run speed is around 600 cm/s (6 m/s)
	const float MaxRunSpeed = 600.0f;
	float SpeedRatio = FMath::Clamp(CharacterSpeed / MaxRunSpeed, 0.0f, 1.0f);
	
	// Apply speed ratio as weight, scaled down to max 50% influence
	// Max 50% ensures auto-orient is assistive, not dominant
	// This allows blending with input modules (R01/R02)
	const float MaxInfluence = 0.5f;
	float Weight = SpeedRatio * MaxInfluence;
	
	// Alternative: Get weight from config
	// float AutoOrientStrength = StateConfig.Module. Rotation.AutoOrientStrength;
	// if (AutoOrientStrength > 0.0f && AutoOrientStrength <= 1.0f)
	// {
	//     Weight = SpeedRatio * AutoOrientStrength;
	// }
	
	// Step 6: Fill output structure
	OutOutput.RotationOutput = DesiredRotation;
	OutOutput.bHasRotationOutput = true;
	
	// Set weight for blending
	// Lower weight = subtle effect (assistive)
	// Higher weight = stronger effect (more noticeable)
	OutOutput.Weight = Weight;
	
	return true;
}
```

**说明：**
- 基于速度向量计算朝向
- 仅控制Yaw，保留Pitch
- 动态权重（0-0.5范围）
- Blend策略（辅助性，不强制）

【技术说明】

**R06的核心机制：**

### 1. 自动对齐的触发条件

**为什么需要这么多条件？**

```
条件1: ! bHasTarget
- 战斗中不要自动对齐
- 让锁定模块控制

条件2: bIsMoving && Speed > MinSpeed
- 静止时不对齐（保持玩家设置的角度）
- 太慢时不对齐（微调位置时）

条件3: InputMagnitude < 0.1
- 玩家控制相机时不干扰
- 只在"无输入"时辅助

结果：
- 仅在探索移动时辅助
- 不干扰战斗和玩家控制
- 自然流畅的体验
```

### 2. 仅控制Yaw的原因

**为什么不控制Pitch？**

```
场景：玩家向前跑，同时向上看天空

如果控制Pitch：
DesiredRotation.Pitch = 0°（水平）
→ 相机被强制拉回水平
→ 玩家无法看天空
→ 体验差

只控制Yaw：
DesiredRotation.Yaw = MovementDirection
DesiredRotation.Pitch = CurrentPitch（保持）
→ 相机水平方向对齐移动
→ 垂直方向保持玩家设置
→ 体验好 ✅
```

### 3. 动态权重系统

**权重计算：**
```
Weight = SpeedRatio * MaxInfluence

SpeedRatio = Clamp(CharacterSpeed / 600, 0, 1)
MaxInfluence = 0.5

示例：
速度100 cm/s (慢走): SpeedRatio=0.17 → Weight=0.085 (8.5%)
速度300 cm/s (快走): SpeedRatio=0.5  → Weight=0.25 (25%)
速度600 cm/s (跑步): SpeedRatio=1.0  → Weight=0.5 (50%)

效果：
- 慢走：轻微对齐
- 快走：适度对齐
- 跑步：明显对齐
```

**为什么最大50%？**
```
如果MaxInfluence = 1.0（100%）：
- R06完全控制相机
- 玩家输入被压制
- 感觉失控

如果MaxInfluence = 0.5（50%）：
- R06提供辅助
- 玩家输入仍主导（R01权重1.0）
- 混合：R01 67% + R06 33%
- 平衡 ✅
```

### 4. 与其他模块的交互

**优先级关系：**
```
R01 (Free Input): 100, Override
R06 (Auto Orient): 90, Blend

场景：玩家移动但不控制相机

如果R01激活（有输入）：
- R06的ShouldActivate返回false（检测到输入）
- 只有R01工作
- 玩家完全控制

如果R01不激活（无输入）：
- R06激活
- 自动对齐生效
- 辅助性调整
```
