```cpp
/**
 * ShouldActivate - Determine if lagged camera input should be active
 * 
 * R02 activates when:
 * - Module is enabled
 * - No target is locked (free camera, not lock-on)
 * - Input is not explicitly ignored (StateConfig flag)
 * - Rotation lag is explicitly enabled in StateConfig
 * 
 * R02 has the same priority as R01, so they are mutually exclusive.
 * The StateConfig flag bEnableRotationLag determines which one activates.
 * 
 * Use Cases:
 * - Cinematic camera movements
 * - Smooth, less responsive controls (some players prefer this)
 * - Reducing motion sickness
 */
bool UCameraModule_R02_PlayerInput_Lagged:: ShouldActivate(
	const FStageExecutionContext& Context, 
	const FCameraStateConfig& StateConfig) const
{
	// Check if module is enabled
	if (!bIsEnabled)
	{
		return false;
	}
	
	// Don't activate when target is locked
	// Lock-on rotation modules (R03-R05) handle that
	if (Context.InputContext.bHasTarget)
	{
		return false;
	}
	
	// Check if input is explicitly ignored (cinematic states, etc.)
	if (StateConfig.StateBase.Flags.bIgnoreInput)
	{
		return false;
	}
	
	// Check if rotation lag is explicitly enabled
	// This is the key difference from R01
	// If bEnableRotationLag is false, R01 will activate instead
	if (!StateConfig.Module.Rotation.bEnableRotationLag)
	{
		return false;
	}
	
	// All conditions met - activate lagged camera control
	return true;
}
```

- 与R01相似的激活条件
- 额外检查bEnableRotationLag标志

**替换为完整实现：**
```cpp
/**
 * Compute - Calculate smoothly lagged rotation from player input
 * 
 * This module provides cinematic smooth rotation using interpolation.
 * Unlike R01 which directly applies input, R02 interpolates towards
 * the target rotation, creating a lagged, smooth effect.
 * 
 * Steps:
 * 1. Initialize if first frame
 * 2. Get camera input from InputContext
 * 3. Calculate target rotation from input (same as R01)
 * 4. Clamp pitch and normalize yaw (same as R01)
 * 5. Smoothly interpolate SmoothedRotation towards TargetRotation
 * 6. Output the smoothed rotation
 * 
 * Key Difference from R01:
 * - R01:  Input directly controls CurrentRotation (immediate response)
 * - R02: Input controls TargetRotation, SmoothedRotation interpolates to it (delayed response)
 * 
 * The interpolation uses RInterpTo for proper rotational interpolation,
 * which handles angle wrapping correctly (e.g., 359° → 1°).
 * 
 * @param Context The current pipeline execution context
 * @param StateConfig The current camera state configuration
 * @param OutOutput Output structure to fill with computed values
 * @return True if computation succeeded
 */
bool UCameraModule_R02_PlayerInput_Lagged::Compute(
	const FStageExecutionContext& Context, 
	const FCameraStateConfig& StateConfig, 
	FModuleOutput& OutOutput)
{
	// Create base output with module identity
	OutOutput = CreateBaseOutput();
	
	// Step 1: Initialize if first frame (safety check)
	// OnActivate should handle this, but double-check for safety
	if (!bInitialized)
	{
		TargetRotation = Context.InputContext.PreviousCameraRotation;
		SmoothedRotation = Context.InputContext.PreviousCameraRotation;
		bInitialized = true;
	}
	
	// Step 2: Get camera input from InputContext
	FVector2D CameraInput = Context.InputContext.CameraInput;
	
	// Step 3: Get sensitivity from state config
	float InputSensitivity = StateConfig. Module.Rotation.InputSensitivity;
	
	// If InputSensitivity not configured, use default
	if (InputSensitivity <= 0.0f)
	{
		InputSensitivity = 1.0f;
	}
	
	// Step 4: Calculate target rotation deltas from input
	// This is identical to R01's input processing
	float DeltaYaw = CameraInput.X * InputSensitivity;
	float DeltaPitch = CameraInput.Y * InputSensitivity * -1.0f;  // Invert pitch
	
	// Step 5: Accumulate deltas to target rotation
	// Input directly controls TargetRotation (not SmoothedRotation)
	TargetRotation.Yaw += DeltaYaw;
	TargetRotation. Pitch += DeltaPitch;
	
	// Step 6: Clamp pitch on target rotation
	float MinPitch = StateConfig.StateBase.Rotation.MinPitch;
	float MaxPitch = StateConfig.StateBase. Rotation.MaxPitch;
	
	// Validate pitch limits are reasonable
	if (MinPitch >= MaxPitch)
	{
		MinPitch = -80.0f;
		MaxPitch = 80.0f;
	}
	
	TargetRotation.Pitch = FMath::Clamp(TargetRotation.Pitch, MinPitch, MaxPitch);
	
	// Step 7: Normalize yaw on target rotation
	TargetRotation.Yaw = FMath::Fmod(TargetRotation.Yaw, 360.0f);
	if (TargetRotation. Yaw < 0.0f)
	{
		TargetRotation.Yaw += 360.0f;
	}
	
	// Step 8: Force roll to zero
	TargetRotation.Roll = 0.0f;
	
	// Step 9: Get rotation lag speed from state config
	float LagSpeed = StateConfig.StateBase.Lag.RotationLagSpeed;
	
	// If LagSpeed not configured or invalid, use default
	if (LagSpeed <= 0.0f)
	{
		LagSpeed = 5.0f;  // Default:  moderate smoothing
	}
	
	// Step 10: Smoothly interpolate SmoothedRotation towards TargetRotation
	// Use RInterpTo for proper rotational interpolation
	// RInterpTo handles angle wrapping correctly (e.g., 359° → 1° goes through 360°, not backwards)
	SmoothedRotation = FMath::RInterpTo(
		SmoothedRotation,      // Current rotation
		TargetRotation,        // Target rotation
		Context.DeltaTime,     // Delta time
		LagSpeed               // Interpolation speed
	);
	
	// Alternative: Use manual interpolation with GetInterpAlpha (same as Position modules)
	// float InterpAlpha = GetInterpAlpha(Context.DeltaTime, LagSpeed);
	// SmoothedRotation. Pitch = FMath::FInterpTo(SmoothedRotation.Pitch, TargetRotation.Pitch, Context.DeltaTime, LagSpeed);
	// SmoothedRotation.Yaw = FMath::FInterpTo(SmoothedRotation.Yaw, TargetRotation. Yaw, Context.DeltaTime, LagSpeed);
	// SmoothedRotation.Roll = 0.0f;
	
	// Step 11: Fill output structure
	OutOutput.RotationOutput = SmoothedRotation;
	OutOutput.bHasRotationOutput = true;
	
	return true;
}
```

**说明：**
- 双状态系统：TargetRotation（输入目标）+ SmoothedRotation（平滑输出）
- 输入处理与R01相同（DeltaYaw/DeltaPitch）
- 使用FMath::RInterpTo进行旋转插值
- RInterpTo正确处理角度环绕（359°→1°）

【技术说明】

**R02的核心机制：**

### 1. 双状态系统

**R01 vs R02：**

| 特性 | R01 (Free) | R02 (Lagged) |
|------|-----------|-------------|
| **状态数量** | 1个（CurrentRotation） | 2个（Target + Smoothed） |
| **输入作用** | 直接修改CurrentRotation | 修改TargetRotation |
| **输出** | CurrentRotation | SmoothedRotation |
| **响应性** | 即时 | 延迟平滑 |

**工作流程：**
```
R01:
Input → CurrentRotation → Output
      (直接)

R02:
Input → TargetRotation → [插值] → SmoothedRotation → Output
      (直接)              (延迟)
```

### 2. RInterpTo的作用

**为什么使用RInterpTo而非简单Lerp？**

```cpp
// 问题：角度环绕
情况：SmoothedRotation.Yaw = 359°, TargetRotation. Yaw = 1°

使用简单Lerp：
Lerp(359, 1, 0.1) = 359 * 0.9 + 1 * 0.1 = 323.2°
→ 相机向左旋转37°（错误！应该向右旋转2°）

使用RInterpTo：
RInterpTo内部处理角度环绕
359° → 360° → 0° → 1°（正确路径）
→ 相机向右旋转2°（正确！）
```

**RInterpTo的优势：**
- ✅ 自动处理角度环绕
- ✅ 选择最短旋转路径
- ✅ 帧率无关
- ✅ 专为FRotator设计

### 3. 与P02的相似性

**Position模块P02 vs Rotation模块R02：**

| 特性 | P02 (Position Lagged) | R02 (Rotation Lagged) |
|------|----------------------|----------------------|
| **输入** | 角色位置 | 玩家输入 |
| **目标** | TargetPosition | TargetRotation |
| **平滑** | CurrentPosition | SmoothedRotation |
| **插值** | FMath:: Lerp (FVector) | FMath::RInterpTo (FRotator) |
| **特殊处理** | 无 | 角度环绕 |

**设计一致性：** ✅ 两者使用相同的平滑策略模式

### 4. 激活条件对比

**R01 vs R02激活条件：**

```cpp
R01:
- bIsEnabled ✅
- ! bHasTarget ✅
- ! bIgnoreInput ✅
- (无额外条件)

R02:
- bIsEnabled ✅
- !bHasTarget ✅
- !bIgnoreInput ✅
- bEnableRotationLag ✅ （额外条件）
```

**互斥关系：**
- 同优先级（100）
- 由bEnableRotationLag决定谁激活
- 配置灵活切换

### 5. LagSpeed的影响

**不同LagSpeed的效果：**

```
LagSpeed = 1.0:
- 非常慢的平滑
- 相机"拖"在后面
- 极度电影化

LagSpeed = 5.0 (默认):
- 适度平滑
- 平衡响应性和平滑度
- 推荐值

LagSpeed = 10.0:
- 快速平滑
- 接近R01的响应性
- 轻微柔和感

LagSpeed = 50.0:
- 几乎即时
- 与R01相似
- 基本无平滑
```
